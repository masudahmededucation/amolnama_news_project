    _save_actor_person()          — Name + Identity → [person].[person]
    _save_actor_occupation()      — Occupation + Institution → [person].[person_job] (get-or-create JobTitle + Organisation)
    _resolve_actor_role()         — Cached lookup: (role_code or role_id) → (status_id, status_code) tuple

    _save_actor_profile()         — Actor Type + Party Details → [investigation].[incident_involved_actor_profile]
                                    Accepts role_code= param for server-side role assignment.
                                    Sets BOTH link_ref_status_incident_involved_actor_role_id (FK)
                                    AND incident_involved_actor_role_group_code (denormalised code).
    _save_actors_from_json()      — Top-level: parse JSON + call both helpers per actor
                                    Uses _FIELD_TO_ROLE mapping:
                                      accused_json → 'ACCUSED' (status_id=34)
                                      victim_json  → 'VICTIM'  (status_id=39)
                                      witness_json → 'WITNESS' (status_id=40)

MODULARISATION RULE — UI + DATA ENTRY ARE IDENTICAL EVERYWHERE:

  The Name Group, Personal Identity Group, and Party Details Group are IDENTICAL
  across all repeaters AND across all rendering modes (template, HTML string, DOM).
  Only the CSS class for event delegation differs.

  This modularisation is NOT just UI rendering — the DATA ENTRY (save operation)
  is also modularised. All forms use the SAME shared Python helper functions:
    _save_actor_person()     → [person].[person]
    _save_actor_occupation() → [person].[person_job] (if occupationId provided)
    _save_actor_profile()    → [investigation].[incident_involved_actor_profile]

  EVERY form that collects actor data goes through these functions.
  The JSON keys, DB columns, save order, and target tables are identical
  everywhere. No form has its own inline Person.objects.create,
  JobTitle/Organisation get-or-create, or IncidentInvolvedActorProfile.objects.create
  for standard actor fields.

  Form-specific extras (WCV marital status/relationship/attributes, July mother name, etc.)
  are handled AFTER the shared save, as post-updates or additional table inserts.

  Forms using shared save helpers:
    Crime & Violence      — _save_actors_from_json (accused, victim, witness)
    Extortion             — _save_actors_from_json (accused, victim, witness)
    Land Grabbing         — _save_actors_from_json (accused, victim, witness)
    Price Hike/Syndicate  — _save_actors_from_json (accused, victim, witness)
    Watchdog (নজরদারি)    — _save_actors_from_json (politician)
    Civic Community       — _save_actors_from_json (accused, victim, witness)
    WCV                   — _save_actor_person + _save_actor_profile (victim + accused)
    July Uprising         — _save_actor_person + _save_actor_profile (martyr)

--------------------------------------------------------------------------------
  INSERT SEQUENCE  (per actor card on form submit)
--------------------------------------------------------------------------------

  1. INSERT INTO [person].[person] (first_name_en, last_name_en, first_name_bn,
        last_name_bn, name_alias_bn, link_gender_id, link_religion_id,
        date_of_birth, age, link_birth_district_id, primary_mobile_number,
        is_active, created_at, modified_at)
     → returns person_id
     Note: name_alias_en = NULL (alias stored in BN column only)

  2. If contact count > 1:
     INSERT INTO [contact].[phone] (link_person_id=person_id, ...)
     for each additional phone.

  3. INSERT INTO [investigation].[incident_involved_actor_profile]
     (link_ref_status_incident_involved_actor_role_id, incident_involved_actor_role_group_code,
      link_form_type_id, link_coll_news_entry_id, link_person_id,
      link_ref_status_incident_involved_actor_type_id, incident_involved_actor_type_details,
      actor_organization_name, actor_designation, actor_patron_name, actor_statement,
      link_ref_status_victim_medical_condition_id, victim_medical_treatment_location_name,
      created_at)
     Values: link_ref_status_incident_involved_actor_role_id → resolved by _resolve_actor_role()
               using role_code from _FIELD_TO_ROLE (accused_json→ACCUSED, victim_json→VICTIM, witness_json→WITNESS)
               or explicit role_code param (WCV, July Uprising)
             incident_involved_actor_role_group_code → denormalised status_code ('ACCUSED'/'VICTIM'/'WITNESS')
             link_form_type_id = form type FK for this news entry
             link_person_id = person_id from step 1
             link_ref_status_incident_involved_actor_type_id = actorTypeId (from JS)
             incident_involved_actor_type_details = actorTypeDetail (from JS)
             link_ref_status_victim_medical_condition_id = conditionId (from JS, crime victim only)
             victim_medical_treatment_location_name = medicalLocation (from JS, crime victim only)

All actor photo uploads (accused / victim / witness) save via:
  _save_or_reuse_asset() → [media].[asset] (dedup by SHA-256)
  → [newshub].[coll_news_asset] (junction, sort_order ranges: accused=100-102, victim=200-202, witness=300-302)

POST fields per actor type:
  Accused  : accused_photos     (FileList, max 3) + accused_photos_descriptions_json
  Victim   : victim_photos      (FileList, max 3) + victim_photos_descriptions_json
  Witness  : witness_photos     (FileList, max 3) + witness_photos_descriptions_json

descriptions_json format: [{"index": 0, "description": "..."}, ...]

SAVE PATTERN — per file in each FileList:
  1. Save file bytes to storage
  2. INSERT INTO [media].[asset] → returns asset_id
  3. INSERT INTO [media].[news_entry_asset]:
       link_asset_id                        = asset_id (from step 2)
       link_coll_news_entry_id              = news entry id
       link_person_id                       = person_id (AFTER person INSERT completes)
                                              NULL for crime evidence files
       link_ref_status_incident_involved_actor_role_id = status_id (see table below)

ROLE ID LOOKUP — [investigation].[ref_status]
  group_code = 'incident_involved_actor_role'
  WHERE status_code = ...

  Actor type        status_code    status_id   link_person_id
  ────────────────  ─────────────  ──────────  ──────────────────────────────
  Accused photos    ACCUSED        (lookup)    accused person_id
  Victim photos     VICTIM         (lookup)    victim person_id
  Witness photos    WITNESS        (lookup)    witness person_id
  Crime evidence    INVESTIGATOR   36          NULL or named officer's person_id

NOTE: status_ids for ACCUSED / VICTIM / WITNESS must be looked up at runtime.
  Only INVESTIGATOR = 36 is a confirmed hardcoded value.

ORDERING CONSTRAINT:
  Person INSERTs (step 1 of actor sequence) must complete before
  news_entry_asset INSERTs so that link_person_id is available.
