TRANSACTION POLICY — ALL FORMS
================================================================================
Function  : _handle_news_submission()  (newshub/views.py)
Updated   : 2026-03-13

ALL database writes for a single form submission are wrapped in ONE
transaction.atomic() block. This guarantees:

  ✅ FULL SUCCESS  — every table row (contributor, news entry, attachments,
                     social sources, tags, form-specific evidence) is committed
                     together.
  ✅ FULL ROLLBACK — if ANY step fails (malformed JSON, DB constraint violation,
                     data type conversion error, file upload error), the ENTIRE
                     transaction is rolled back. No orphan rows. No partial data.

Implementation:
  - Outer block: try → with transaction.atomic(): → except (...)
  - All inner try/except blocks re-raise exceptions (no silent `pass`).
    If JSON parsing or data conversion fails for submitted data, the exception
    propagates to the outer handler, triggering a full rollback.
  - Outer except catches: IntegrityError, DatabaseError, json.JSONDecodeError,
    ValueError, TypeError, IndexError.
  - On failure, the user sees an error message and can re-submit.

Rules:
  1. NEVER add `except (...): pass` inside the transaction block.
     Always use `raise` to propagate errors.
  2. The `if raw_json:` guard handles "no data submitted" (skips the block).
     The try/except handles "data was submitted but is invalid" (raises → rollback).
  3. File uploads (attachments, photos) also fail-fast — no silent skip.
  4. Date helpers (_sdate/_sedate) return None for empty strings but let
     ValueError propagate for malformed dates — no silent `return None` on error.
  5. Any new form-specific save block MUST follow this pattern.

Audit (2026-03-13, round 4):
  - ALL except blocks inside the transaction and its helper functions use `raise`.
  - ZERO `pass` blocks exist anywhere in the file (verified by grep).
  - Helper functions also fixed:
    • _save_evidence_files(): descriptions JSON parse error → raise (was pass)
    • _save_actor_person(): malformed DOB → raise (was pass, now no try/except)
    • _save_actors_from_json(): malformed actor JSON → raise (was pass)
  - Outer except catches: IntegrityError, DatabaseError, json.JSONDecodeError,
    ValueError, TypeError, IndexError.
  - Forms covered: crime, extortion, land grabbing, civic, price hike, political
    tracker, global news, war/conflict, sports, entertainment, July uprising,
    women/child violence + common (attachments, social sources, tags, actors).

CLIENT-SIDE VALIDATION — ref_asset_type rules (✅ CONFIRMED 2026-03-05)
  Source  : [media].[ref_asset_type]   Django model: RefAssetType  (multimedia/models.py)
  Context : asset_type_rules_json → <script type="application/json" id="asset-type-rules-data">
  JS reads : mime_type, max_size_kb, allowed_extension   (only active rows)

  Validation order in news-attachment-upload.js:
    1. Match uploaded file by mime_type first
    2. Fallback: match by allowed_extension (single extension per row, e.g. ".jpg")
    3. No rule matched → reject: "এই ধরনের ফাইল অনুমোদিত নয় (File type not allowed)"
    4. file.size > max_size_kb × 1024 → reject with max MB message

  ref_asset_type columns used:
    [mime_type]          [nvarchar](100) NULL   — e.g. "image/jpeg"
    [max_size_kb]        [int] NOT NULL         — e.g. 5120 (= 5 MB)
    [max_size_mb]        [decimal](18,2) NULL   — informational only (JS uses max_size_kb)
    [allowed_extension]  [nvarchar](50) NULL    — e.g. ".jpg"
    [is_active]          [bit] NOT NULL         — inactive rows are excluded from rules

SERVER-SIDE SAVE — two-table flow (✅ CONFIRMED 2026-03-05)

  Step A — _save_or_reuse_asset() helper:
    1. Compute SHA-256 hash of uploaded file
    2. Look up [media].[asset] by hash_sha256 + file_size_bytes + is_active=True
    3. If found    → reuse existing asset (get-or-create rule — confirmed)
    4. If not found → INSERT into [media].[asset]:
         [asset_guid]           → uuid4()
         [file_original_name]   → uploaded_file.name
         [file_extension]       → os.path.splitext(name)[1].lower()
         [file_mime_type]       → uploaded_file.content_type
         [file_size_bytes]      → uploaded_file.size
         [asset_description_bn] → per-file description from attachment_descriptions_json (or NULL)
         [hash_sha256]          → SHA-256 digest (binary)
         [hash_algorithm_used]  → 'SHA-256'
         [hash_is_verified]     → True
         [hash_last_verify_at]  → now
         [is_active]            → True
         [created_at]           → now
         [modified_at]          → now
         NOTE: [file_storage_path] is a SQL Server COMPUTED column — never defined in Django model.
               After INSERT, read back via raw SQL: SELECT file_storage_path WHERE asset_id = %s
               Then save physical file to MEDIA_ROOT / file_storage_path.
    5. Returns Asset instance

  Step B — Create junction row in [newshub].[coll_news_asset]:
    Django model: CollNewsAsset  (newshub/models.py)

    [link_coll_news_entry_id]       → entry.coll_news_entry_id
    [link_asset_id]                 → asset.asset_id  (from Step A)
    [coll_news_asset_caption_bn]    → attachment_caption_bn  (first file only; NULL for others)
    [is_featured]                   → True if this file's index == featured_file_index
    [sort_order]                    → loop index (0-based)
    [created_at]                    → now

  Max files per submission: 4

SOCIAL SOURCES (repeater — present on all forms)
-------
POST key  : social_source_json  (JSON array, news-social-source-repeater.js)
Repeater  : multiple entries allowed per submission

Platform dropdown source:
  TABLE  : [media].[ref_social_media_platform_type]
  Django : RefSocialMediaPlatformType  (newshub/models.py)
  Context key : platform_types
  Template value : pt.social_media_platform_type_id

  Step A — get-or-create URL record in [media].[social_media_url_library]:
    Django : SocialUrlLibrary  (multimedia/models.py)
    Filter by social_url == src_url  (DB also has unique constraint on social_url_hash = SHA2_256 of lowercased url)
    If not found → INSERT:
      [link_social_media_platform_type_id] → platformId from JSON (int)
      [social_url]                         → url from JSON
      [social_embed_code]                  → embedCode from JSON (or NULL)
      [created_at]                         → now
      NOTE: [social_url_hash] is a computed persisted column (SHA2_256) — never written by Django

  Step B — create junction row in [newshub].[coll_news_social_media_source]:
    Django : CollNewsSocialMediaSource  (newshub/models.py)
    One row per social source entry (multiple rows per news entry allowed)
    [link_coll_news_entry_id]          → entry.coll_news_entry_id
    [link_social_media_url_library_id] → url_record.social_media_url_library_id
    [created_at]                       → now

  JSON keys from news-social-source-repeater.js:
    platformId   →  link_social_media_platform_type_id
    url          →  social_url
    embedCode    →  social_embed_code

INVOLVED ACTOR PROFILE (multi-person repeater — used by forms with a person/actor step)
-------
POST key  : involved_actors_json  (JSON array serialized by JS repeater)
Repeater  : multiple entries per submission — one row per person
Django    : IncidentInvolvedActorProfile  (investigation/models.py)
TABLE     : [investigation].[incident_involved_actor_profile]

Save flow — per actor entry:

  Step A — INSERT new Person row into [person].[person]:
    Rule: Person → ALWAYS INSERT a new row. Never check for existing (confirmed).
    → get back person_id

  Step A2 (optional) — If the form collects profession/occupation:
    Shared helper: _save_actor_occupation(person_id, actor_data, now, occ_key='occupationId')
    Django models: JobTitle (person/models.py), PersonJob (person/models.py)
    Does nothing if occupationId == 0 or institution is blank.

    Internally:
    1. Get-or-create [career].[job_title] by job_title_name_bn / job_title_name_en
       → get job_title_id
    2. Get-or-create [directory].[organisation] by name
       → infer organisation_type_id via _infer_org_type_id(name) keyword matching
       → get organisation_id
    3. INSERT [person].[person_job]:
         [link_person_id]       → person_id from Step A
         [link_job_title_id]    → job_title_id from step 1
         [link_organisation_id] → organisation_id from step 2
         [link_department_id]   → NULL unless form collects department
         [start_date]           → today (or form-supplied date)
         [is_active]            → True
         [created_at]           → now

    Used by: WCV victim (Step 4), WCV accused (Step 6), July Uprising martyr (Step 4)
    Note: occ_key='occupation' for July Uprising (different JSON key name)

  Step B — INSERT one row into [investigation].[incident_involved_actor_profile]:
    [link_coll_news_entry_id]                              → entry.coll_news_entry_id
    [link_person_id]                                       → person_id from Step A
    [link_person_marriage_id]                              → marriage_id if form collects it, else NULL
    [link_ref_status_incident_involved_actor_role_id]      → status_id from [investigation].[ref_status]
                                                              group_code = 'incident_involved_actor_role'
                                                              ACCUSED=34, VICTIM=39, WITNESS=40
                                                              Set via _FIELD_TO_ROLE or explicit role_code param.
    [incident_involved_actor_role_group_code]              → denormalised status_code string
                                                              'ACCUSED', 'VICTIM', 'WITNESS', etc.
                                                              Enables querying by role without join.
    [link_ref_status_incident_involved_actor_type_id]      → actor type status_id from JS (actorTypeId)
    [incident_involved_actor_type_details]                 → actor type detail text from JS
    [actor_organization_name]                              → from JS (organization)
    [actor_designation]                                    → from JS (designation)
    [actor_patron_name]                                    → from JS (patron)
    [actor_statement]                                      → from JS (statement)
    [link_ref_status_victim_medical_condition_id]          → from JS (conditionId, crime victim only)
    [victim_medical_treatment_location_name]               → from JS (medicalLocation, crime victim only)
    [link_form_type_id]                                    → looked up from [newshub].[ref_news_form_type]
                                                              by group_code (same ID used in coll_news_entry)
    [created_at]                                           → DB default (sysdatetime())

  Role assignment (ALWAYS set — every actor_profile row gets both FK and denormalised code):
    _save_actors_from_json() uses _FIELD_TO_ROLE mapping:
      accused_json → role_code='ACCUSED' → _resolve_actor_role() → (34, 'ACCUSED')
      victim_json  → role_code='VICTIM'  → _resolve_actor_role() → (39, 'VICTIM')
      witness_json → role_code='WITNESS' → _resolve_actor_role() → (40, 'WITNESS')
    WCV + July Uprising pass role_code directly to _save_actor_profile().

  Notes:
    - Each actor in the repeater creates exactly one Person row + one actor_profile row.
    - Multiple rows per news entry are allowed (one per actor added).
    - link_person_marriage_id is NULL unless the form specifically collects spouse/marriage data.
    - Profession is stored in [person].[person_job] (not on the person row itself).
    - Involvement type dropdown populated from ref_status filtered by
      group_code = 'incident_involved_actor_role'.

KEY CODING RULES (apply to ALL forms)
================================================================================

1. _bn columns → display/storage only. Never use in order_by(). Never filter()
   on _bn alone. Search: Q(col_en__icontains=q) | Q(col_bn__icontains=q)

2. Non-mandatory text → save as None (not "").
   Pattern: (val or '').strip() or None

3. BIT columns with DB DEFAULT ((0)) → do NOT add null=True to Django field.
   Do NOT add default= to any model field — DB handles its own defaults.

4. ref_status bulk code lookup:
   a) Collect all IDs into a set, remove zeros
   b) code_map = {row['status_id']: row['status_code'].upper()}
      via RefStatus.objects.filter(status_id__in=all_ids).values(...)
   c) Derive column name from code or compare against known codes.

5. Tom Select dropdowns → never set .value directly.
   Use: element.tomselect.setValue(value, true)
   Then dispatch native 'change' event for cascade triggers.

6. SQL Server computed columns → NEVER define as Django model fields.
   Leave a comment in the model. Example: # hash_headline_check (computed)

7. INSERT vs GET-OR-CREATE rule (confirmed 2026-03-05):
   Person          → always INSERT new row. Never check for existing.
   Everything else → get-or-create: check first, insert if not found, use the ID.

   Applies to (non-exhaustive):
     organisation           → get-or-create by name → use link_organisation_id
     education institution  → get-or-create by name → use for person_education
     media asset (file/pic) → get-or-create in [media].[asset] by file hash or path
                               if not found → INSERT new asset row, get asset_id
                               if found     → use existing asset_id

8. link_X_id column naming convention:
   Strip 'link_' prefix and '_id' suffix to get the table name.
   Example: link_organisation_id → [organisation] table
            link_job_title_id    → [career].[job_title] table
            link_gender_id       → [person].[gender] table
   'ref_' is a table naming prefix in the DB schema — it does NOT appear
   in link column names. link_issue_sub_type_id → [ref_issue_sub_type] table.

EDIT MODE (added 2026-03-24)
================================================================================

When ?edit=<coll_news_entry_id> is in GET params:

1. View fetches existing CollNewsEntry + related data
2. helpers.py:build_edit_data() reconstructs all form JSON from DB:
   - _build_shared_edit_data(entry) — contributor, news content, location, social, tags
   - _build_actor_edit_data(entry_id) — bulk Person fetch (no N+1)
   - _build_extortion_edit_data(entry_id) — BIT flag → status_id reverse mapping
3. JSON passed to template as edit_data_json context variable
4. news-form-edit-load.js runs BEFORE component scripts:
   - Sets hidden JSON inputs (#accused-json, #extortion-incident-json, etc.)
   - Sets window.__EDIT_MODE__ = true
   - Components call restoreFromHiddenInput() on init, find the data
5. news-form-persist.js skips localStorage restore when __EDIT_MODE__ is true
6. On submit: _handle_news_submission() detects edit_entry_id in POST:
   - UPDATE existing entry (not CREATE)
   - DELETE + re-INSERT junction tables (tags, actors, form-specific)
   - Skip duplicate headline check for own entry
   - Redirect to article view page

Performance: 11 RefStatus queries batched into 1. N+1 actor/social queries eliminated.
