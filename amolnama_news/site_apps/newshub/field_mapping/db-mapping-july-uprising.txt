JSON keys from news-july-uprising-martyr.js:

  Group 1 — Name (via window.newshubPersonName):
  firstNameEn   → person.first_name_en      (NOT NULL — required)
  lastNameEn    → person.last_name_en        (NOT NULL — required)
  firstNameBn   → person.first_name_bn
  lastNameBn    → person.last_name_bn
  slug          → (informational only; auto-generated from EN name — NOT saved to DB)
  alias         → person.name_alias_bn       (from person-name-alias-fields.html)

  Group 2 — Universal Personal Identity (same 6 fields as crime repeaters):
  genderId      → person.link_gender_id      (INT FK, from <select>, ref_status group='gender')
  religionId    → person.link_religion_id    (INT FK, from <select>, ref_status group='religion')
  ✅ FIXED: JS keys changed from `gender`/`religion` → `genderId`/`religionId` to match
            what _save_actor_person() reads. Old keys caused silent NULL saves.
  age           → person.age                 (INT, nullable)
  dob           → person.date_of_birth       (DATE, nullable, max=2024-08-07)
  districtId    → person.link_birth_district_id (INT FK to [location].[district])
                  ✅ Now a <select> dropdown (was free text). Sends district_id directly.
                  View context: july_districts queryset.
  contact       → person.primary_mobile_number (VARCHAR, nullable)

  Group 3 — Occupation & Workplace (extra fields not in universal identity):
  occupation    → status_id from ref_status (group_code='july2024_occupation_list')
                  → get-or-create [career].[job_title] by status_name_bn/en
                  → person_job.link_job_title_id
  institution   → text → get-or-create [directory].[organisation] by name
                  → person_job.link_organisation_id

  Group 4 — Family:
  fatherFirstName → person.father_first_name_bn  (from Name module, person-name-fields.html)
  fatherLastName  → person.father_last_name_bn   (from Name module, person-name-fields.html)
  motherFirstName → person.mother_first_name_bn  (from person-family-name-fields.html)
  motherLastName  → person.mother_last_name_bn

  Group 5 — Home Address (via window.julyMartyrHomeLocation):
  homeDistrictId    → july2024_fact_protest.link_home_district_id       (INT NULL)
  homeDistrictName  → (display only — not saved)
  homeUpazilaId     → july2024_fact_protest.link_home_upazila_id        (INT NULL)
  homeUpazilaName   → (display only — not saved)
  homeLocalBodyId   → july2024_fact_protest.link_home_union_parishad_id (INT NULL)
  homeLocalBodyName → (display only — not saved)
  homeWardId        → july2024_fact_protest.link_home_ward_id           (INT NULL)
  homeWardName      → (display only — not saved)
  (no JS field)     → july2024_fact_protest.home_local_area_name        (NVARCHAR(200) NULL — no UI yet)

  Other:
  status        → status_id from ref_status group_code='july2024_victim_current_status'
                  (radio in Step 5 — read in Step 4 serializer; see Step 5 note below)

SAVE FLOW (Step 4 — per martyr):
  Uses shared helpers: _save_actor_person() + _save_actor_occupation() + _save_actor_profile()
  See db-mapping-actor-repeaters.txt for full shared save protocol.
  Mother name is a July-specific post-update after the shared save.

  A. _save_actor_person() → INSERT [person].[person]  (ALWAYS INSERT new row)
       [first_name_en]            ← firstNameEn (required)
       [last_name_en]             ← lastNameEn (required, '' if blank)
       [first_name_bn]            ← firstNameBn
       [last_name_bn]             ← lastNameBn
       [name_alias_bn]            ← alias or None
       [link_gender_id]           ← genderId (int FK, 0 → None)
       [link_religion_id]         ← religionId (int FK, 0 → None)
       [age]                      ← age (or None)
       [date_of_birth]            ← dob (or None)
       [link_birth_district_id]   ← districtId (int FK, 0 → None) — now direct from <select>
       [primary_mobile_number]    ← contact or None
       [father_first_name_bn]     ← fatherFirstName or None
       [father_last_name_bn]      ← fatherLastName or None
       [mother_first_name_bn]     ← motherFirstName or None
       [mother_last_name_bn]      ← motherLastName or None
       [modified_at]              ← now()
     → get back person_id

  A2. _save_actor_occupation(person_id, jm, now, occ_key='occupation')
       → Shared helper: get-or-create JobTitle + Organisation → INSERT person_job
       → Does nothing if occupation == 0 or institution is blank
       (see db-mapping-common-steps.txt Step A2 for full detail)

  B. _save_actor_profile(role_code='VICTIM') → INSERT [investigation].[incident_involved_actor_profile]:
       [link_coll_news_entry_id]                          ← entry.coll_news_entry_id
       [link_person_id]                                   ← person_id from Step A
       [link_person_marriage_id]                          ← NULL (not collected here)
       [link_ref_status_incident_involved_actor_role_id]  ← 39 (VICTIM, resolved by _resolve_actor_role)
       [incident_involved_actor_role_group_code]          ← 'VICTIM' (denormalised role code)
       [group_code]                                       ← 'july_uprising_2024'
       [link_form_type_id]                                ← FK from [newshub].[ref_news_form_type]
                                                            by group_code='july_uprising_2024'
       [created_at]                                       ← DB default (sysdatetime())

NOTE: The victim 'status' radio (step=5) is read by the martyr JS serializer and
      included in july_martyr_json. However it maps to July2024FactProtest
      (link_ref_status_victim_current_status_id) — see STEP 5 below.

Context vars supplied by view:
  july_forces_involved → ref_status filtered by group_code='july2024_protest_suppression_force_involved'

JSON keys from news-july-uprising-oppressors.js:
  forces             → array of status_ids (ints) for checked force checkboxes
  unitBadge          → free text string
  commandingOfficer  → free text string
  ocDc               → free text string
  directives         → free text string (textarea)

SAVE LOGIC — BIT columns (hardcoded):
  The ref_status IDs are ONLY used to populate the UI checkboxes.
  The status_id values in the forces array are NOT saved as FKs.
  The view has a hardcoded mapping from status_code → BIT column name.
  When a checkbox is checked, the corresponding BIT column is set to 1:
    status_code 'POLICE'               → force_involved_is_police = True
    status_code 'RAB'                  → force_involved_is_rab = True
    status_code 'BGB'                  → force_involved_is_bgb = True
    status_code 'ARMY'                 → force_involved_is_army = True
    status_code 'DB_POLICE'            → force_involved_is_db_police = True
    status_code 'BCL'                  → force_involved_is_bcl = True
    status_code 'JUBO_LEAGUE'          → force_involved_is_jubo_league = True
    status_code 'UNKNOWN_PLAINCLOTHES' → force_involved_is_unknown_plainclothes = True
  All unchecked BIT columns default to False (DB DEFAULT 0).
  No FK/link_id column is written for force types.

DB columns on [investigation].[july2024_fact_protest]:
  [force_involved_is_police]               ← True if POLICE in forces else False
  [force_involved_is_rab]                  ← True if RAB in forces else False
  [force_involved_is_bgb]                  ← True if BGB in forces else False
  [force_involved_is_army]                 ← True if ARMY in forces else False

OVERALL SAVE SEQUENCE
------
All data for one submission is written in a SINGLE TRANSACTION:

1. INSERT [newshub].[coll_news_entry]
   - headline_bn, summary_bn, news_date, link_form_type_id, etc.
   - ⚠ link_issue_sub_type_id NOT saved here — column does not exist in coll_news_entry
   → returns coll_news_entry_id

2. INSERT [newshub].[coll_news_entry_tag]        (Step 11 — one row per tag)

3. INSERT [person].[person]                       (Step 4 — martyr identity)
   → returns person_id

4. INSERT [person].[person_job]   (Step 4 — if occupation/institution provided)

5. INSERT [investigation].[incident_involved_actor_profile]  (Step 4 — martyr profile)

6. INSERT [investigation].[july2024_fact_protest]
   All columns written in one INSERT covering Steps 3, 5, 6, 7, 8:
     link_coll_news_entry_id              ← coll_news_entry_id (Step 1)
     link_ref_status_incident_type_id               ← july_sub_type (Step 3)
     link_ref_status_protest_scale_id               ← july_context_json.scale (Step 3)
     link_ref_status_internet_status_id             ← july_context_json.internetStatus (Step 3)
     link_ref_status_curfew_status_id               ← july_context_json.curfewStatus (Step 3)
     link_ref_status_victim_current_status_id       ← july_martyr_status (Step 5)
     victim_story_* fields               ← july_story_json (Step 5)
     victim_story_dependents_count       ← july_story_json.dependents (Step 5)
     victim_medical_* / victim_evidence_* fields  ← july_cause_json (Step 6)
     force_involved_* / force_details_* fields    ← july_oppressors_json (Step 7)
     link_ref_status_verification_status_id / verification_* fields ← july_evidence_json (Step 8)
     verification_eyewitness_count       ← july_evidence_json.eyewitnessCount (Step 8)
     verification_memorial_reference     ← july_evidence_json.memorialRef (Step 8)
     created_by_user_id                  ← request.user.id

7. INSERT [newshub].[coll_news_asset]        (attachments — if any)
8. INSERT [media].[asset]                    (attachment files — get-or-create by hash)
9. INSERT [newshub].[coll_news_social_media_source]  (social sources — if any)
10. INSERT [media].[social_media_url_library]        (social URLs — get-or-create)
