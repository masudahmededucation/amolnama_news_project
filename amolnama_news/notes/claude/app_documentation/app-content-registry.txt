================================================================================
APP: CONTENT — Master Content Registry + Blog Content Type Registry
================================================================================

Schema: [content]
Django App: site_apps/content/
Role: Universal content index for ALL content (posts + blog). Single source of
      truth for content identity, categories, subcategories, AND the runtime
      metadata registry that lets shared code treat every blog content type
      uniformly (title, slug, summary, URL, counts, is_published).

This doc covers TWO separate concerns that live in this app:

  1. [content].[content_registry] DB table  — universal content ID for every
     piece of content (post + blog). Row-level.

  2. content/content_type_registry.py       — Python-side registry that tells
     shared code which fields play which role on each content type. Type-level.

The two concerns work together: content_registry knows what is published
(row), content_type_registry knows how to read a title/slug/count off any
content row (type).

    data (e.g. Bengali season months for blog_bangladesh_season)
  - group_code: identifies which app/filter group the subcategory belongs to
    Current group_codes (Phase 2 added media + season):
      blog_art_category
      blog_poem_category
      blog_stories_category
      blog_stories_age_group
      blog_bangladesh_destination_category

      blog_bangladesh_season               (added 2026-04-11 Phase 2, replaces [blog_bangladesh].[ref_season])
      blog_debate_category
      newshub_topic

[content].[content_registry]
  - PK: content_registry_id (BIGINT IDENTITY)
  - FK: link_content_ref_content_category_id → ref_content_category
  - FK: link_content_ref_content_subcategory_id → ref_content_subcategory
  - FK: link_newshub_ref_news_form_type_id → newshub.ref_news_form_type (newshub only)
  - Universal content ID for ALL content (posts + blog)
  - Denormalized display fields: title_bn/en, slug, summary_bn, content_url, cover_image_url
  - Status: is_published, is_active, published_at
  - Filtered indexes: blog-only (category != post), posts-only, blog-by-user

CONTENT TYPE REGISTRY (content/content_type_registry.py)  — added 2026-04-11 Phase 3b
================================================================================

Purpose
-------
Runtime metadata registry for every blog content type. Single Python dict
CONTENT_TYPE_REGISTRY holds, per content_type_code, everything a shared
consumer needs to know:

  code, label_bn, badge, color, cta,
  model_module, model_name, table_schema, table_name, pk_field,
  title_bn_field, title_en_field, slug_field,
  summary_bn_field, summary_en_field,
  is_published_field, is_active_field, is_featured_field,
  status_code_field,
  bookmark_count_field, like_count_field, view_count_field, comment_count_field,
  detail_url_prefix, category_group_code, secondary_group_code

Adding a new blog content type = ONE entry in this dict. Zero edits in
shared consumers. Portal dashboard, social profile feed, bookmarks,
presentation metadata all pick it up automatically.

Helpers (import from content.content_type_registry)
---------------------------------------------------
  get_spec(code)                  - raw dict for a content_type_code
  iter_all_specs()                - (code, spec) iterator over every type
  get_model_class(code)           - lazy-import the Django model class
  get_code_for_instance(row)      - reverse lookup: model instance -> code

  get_title(row, lang='bn')       - title with bn/en fallback
  get_summary(row, lang='bn')     - summary with bn/en fallback
  get_slug(row)                   - slug_field value, '' if slugless
  get_pk(row)                     - PK value
  get_detail_url(row)             - /prefix/<slug>/ or /prefix/<pk>/ fallback
  get_is_published(row)           - bool, reads real column OR @property uniformly
  get_counts(row)                 - {bookmark, like, view, comment}
  get_presentation(code)          - {badge, color, cta, label_bn}

Currently registered content types (6):
  art, story, poem, destination, media, debate

Not registered (deliberately):
  - newshub / PubArticle — split schema doesn't fit the shared shape
  - post — different app (post/coll_post), not a blog content type

is_published: real columns vs @property
----------------------------------------
Stories + art store is_published as a real BIT NOT NULL column.
Poem/destination/media use string status codes (poem_status_code etc.);
the DB has a PERSISTED computed column that reads
  CASE WHEN <status_code> = 'published' THEN 1 ELSE 0 END.
The computed column CANNOT be declared as a Django model field (SQL Server
rejects writes, Django ORM would include it in INSERT/UPDATE). Instead,
the models.py on poem/destination/media exposes is_published as a Python
@property that reads status_code. Shared code calls get_is_published(row)
and gets the same answer regardless of mechanism.

Extended properties — cross-reference in the DB
------------------------------------------------
notes/sql-script/phase3b-content-type-extended-properties.sql is
auto-generated from the Python registry. Running it stamps a
'registry_role' extended property on every registered column with the
value '<content_type>:<role_key>' (e.g. 'art:title_bn_field').
Inspect from SSMS:

    SELECT OBJECT_SCHEMA_NAME(ep.major_id) AS schema_name,
           OBJECT_NAME(ep.major_id)        AS table_name,
           c.name                           AS column_name,
           CAST(ep.value AS NVARCHAR(500)) AS registry_role
    FROM sys.extended_properties ep
    JOIN sys.columns c
         ON c.object_id = ep.major_id AND c.column_id = ep.minor_id
    WHERE ep.name = N'registry_role'
    ORDER BY schema_name, table_name, column_name;

Idempotent — re-running the script updates properties in place. Do this
after any edit to CONTENT_TYPE_REGISTRY so the DB cross-reference stays
current.

CONTENT CREATE FLOW
================================================================================

1. User creates content in app-specific form
2. App INSERT into source table → gets source PK
3. App calls get_unified_subcategory_id() → gets unified subcategory ID
4. App sets link_content_ref_content_subcategory_id on source table
5. App calls register_content() → gets content_registry_id
6. App sets link_content_registry_id on source table
7. Newsengine brain modules receive content_registry_id for classification/fact-check

================================================================================
CATEGORY NAMING CONVENTION
================================================================================

Every ref category table MUST have:
  - {entity}_code — lowercase, no spaces, for code comparisons
  - {entity}_name_en — Title Case, for English display
  - {entity}_name_bn — Bengali, for Bengali display

All comparisons use _code (lowercase). All display uses _name_en or _name_bn.
Never compare against _name_en.

================================================================================
PK/FK NAMING CONVENTION
================================================================================

PK: {schema}_{table}_id (e.g., blog_art_coll_artwork_id)
FK: link_{schema}_{table}_id (e.g., link_blog_art_coll_artwork_id)
Content FK: link_content_ref_content_subcategory_id (self-documenting)
Content FK: link_content_registry_id (universal content ID)
