QUIZADMIN APP — IMPLEMENTATION PLAN
=====================================

Purpose: Staff-only dashboard that makes reviewing AI-generated quiz
questions fast, accurate, and auditable. Primary consumer of mastermind
engine; no display-to-end-user logic lives here.

Not a replacement for Django admin — complements it. Django admin stays
authoritative for raw CRUD. Quizadmin owns the reviewer workflow.


## GOLD-STANDARD RESEARCH
=========================

Benchmarked tools for AI-generated-content review:

1. Hugging Face annotation tools (Argilla, Label Studio):
   - Side-by-side source + output + confidence scores
   - Keyboard shortcuts (j/k/a/r)
   - Pagination with infinite scroll
   - Filter rail on the left
   - One-keystroke decisions

2. Linear (issue triage):
   - Dense lists, compact rows
   - Filters as chip-style dropdowns in header
   - Bulk select with shift-click
   - Inline state transitions

3. Grafana (dashboard):
   - Metric cards in a responsive grid
   - Color-coded by status (pending=amber, published=green, rejected=red)
   - Drill-down from card → filtered list page

4. Django admin (already in place):
   - Authoritative for edit/delete, bulk actions, raw filtering
   - We LINK to it rather than rebuild it

Takeaways baked into this plan:
- Review Queue is THE workflow. Optimise for density + keyboard speed.
- Dashboard shows metrics Django admin hides (pending review = first metric).
- All per-row destructive operations must show undo within 5 seconds.
- Everything the user can do via UI must also be doable via Django admin
  (one source of truth; admin is the fallback).


## PAGES
========

1. Dashboard   /quizadmin/
   Purpose: at-a-glance health of the question bank + generation pipeline.
   Metric cards:
     - Pending review (amber, highest-priority card, always first)
     - Published questions (green)
     - Draft questions (grey)
     - Rejected questions (red, collapsed by default)
     - AI-generated total (count across all statuses)
     - Books ingested (book count with >=1 chunk)
     - Acceptance rate last 7 days (published / total reviewed)
   Recent generation jobs (last 10, newest first): book, chunks processed,
     created, rejected, duration, status badge.
   CTA button: "Start reviewing" → /quizadmin/review/
   Link to Django admin for power operations.

2. Review Queue   /quizadmin/review/
   Purpose: sequentially review pending AI questions with full context.
   Top bar: pending count, filters (topic, confidence, book, verdict),
     sort (oldest/newest/confidence asc/desc).
   Main area: one question card at a time (NOT a long list — focused view).
   Question card shows:
     - Question text (Bengali)
     - Source chunk text with the answer highlighted
     - Options (ক/খ/গ/ঘ) with correct option marked
     - NLI scores: similarity, entailment, contradiction (traffic-light)
     - Confidence badge: high / medium / low
     - Verdict code: pass_exact_substring / pass_word_match / pass_similarity
       / pass_nli
     - Source citation: book → chapter → page number
   Actions:
     - Approve (keyboard: A)
     - Reject with optional reason (keyboard: R)
     - Skip / next (keyboard: J, right-arrow)
     - Previous (keyboard: K, left-arrow)
     - Edit in Django admin (link)
     - Bookmark for later (keyboard: B)
   URL holds the current question id — refresh-safe, deep-linkable.

3. Books   /quizadmin/books/
   Purpose: overview of ingested books + per-book question counts.
   Table columns: title, pages ingested, chunks, AI questions (total/review/
     published), last generation job, actions (→ review queue filtered by
     this book, → Django admin).

4. Generation Jobs   /quizadmin/jobs/
   Purpose: audit trail of every ingest + generation run.
   Table columns: book, topic, prompt template, model, chunks, created,
     rejected, duration, status, started_by.
   No actions — read-only audit view.

Optional future pages (NOT in this pass):
   - /quizadmin/analytics/ : acceptance rate trends, per-topic accuracy
   - /quizadmin/topics/   : topic-level stats
   - /quizadmin/books/<id>/ingest/ : upload + ingest UI (right now via
     Django admin + shell scripts)

## VIEW SIGNATURES
==================

views.py
  @staff_member_required
  def dashboard_page(request) -> HttpResponse
    Reads: CollQuestion counts by status; CollGenerationJob last 10;
    returns context with metric numbers + recent jobs.

  @staff_member_required
  def review_queue_page(request) -> HttpResponse
    GET params: topic_id, book_id, confidence_level, verdict_code, question_id
    If question_id given, show that one; otherwise next pending by filter.
    Returns full context: question + options + chunk + NLI scores +
    prev/next ids for keyboard nav.

  @staff_member_required
  def books_page(request) -> HttpResponse
    Reads: RefBook list with per-book counts (chunks, questions by status).

  @staff_member_required
  def generation_jobs_page(request) -> HttpResponse
    Reads: CollGenerationJob paginated (50 per page).

views_api.py
  @staff_member_required @require_POST
  def api_approve(request) -> JsonResponse
    Body: {question_id}
    Returns: {success, next_question_id} for chaining.
    Calls: mastermind.ai_generator.approve_question(...)

  @staff_member_required @require_POST
  def api_reject(request) -> JsonResponse
    Body: {question_id, reason_code?}
    Returns: {success, next_question_id}
    Calls: mastermind.ai_generator.reject_question(...)

  @staff_member_required @require_POST
  def api_skip(request) -> JsonResponse
    Body: {question_id}
    Returns: {next_question_id}
    No DB mutation; purely returns next id.

All three reuse existing mastermind APIs where possible — no duplicate logic.

## DESIGN TOKENS + CONVENTIONS (enforce in every file)
======================================================

- Extend core/base.html. No new <html>/<head>.
- External CSS only — no internal <style> blocks, no inline style=.
- All font sizes from --text-xs through --text-5xl. Zero hardcoded rem/px.
- All border-radius from --radius-*.
- All shadows from --elevation-*.
- Every interactive element: :focus-visible { outline:2px solid var(--accent); outline-offset:2px; }
- Every clickable: min-height:var(--touch-target); touch-action:manipulation.
- Every CSS file ends with @media (prefers-reduced-motion:reduce) block.
- Every <img>: loading="lazy" unless above-fold (fetchpriority="high").
- Every <script src>: defer.
- All HTML class names: .quizadmin-component-element pattern.
- NO popup alert()/confirm()/prompt(). Inline error/success bars.
- NO .upper()/.toUpperCase(). Use CSS text-transform:uppercase.
- All form elements: id + name attributes.


## SECURITY
===========

- All page views: @staff_member_required (Django built-in).
- All API views: @login_required + @require_POST + explicit is_staff check.
- CSRF: use Django's default csrf_token in forms; fetch uses
  X-CSRFToken header via window.getCsrfTokenValue() from core/csrf-token.js.
- No raw SQL. ORM only.
- No user-submitted HTML rendering without core.utils.sanitize_user_html()
  (N/A for this app — we only render existing DB strings; they came from
  OCR/LLM which is already considered trusted-but-not-safe, so we always
  Django-escape via default autoescape).
- Rejection reason: if we ever add a free-text field, sanitize it.


## PERFORMANCE
==============

- Dashboard: one SELECT per metric card. Cache 60 seconds in Django cache
  (memcache on prod). Invalidate on new question insert (signal handler
  deferred — not in this pass).
- Review queue: one question at a time; prefetch next 3 question IDs so
  "J = next" is instant (proactive prefetch per CLAUDE.md).
- Books/jobs pages: server-side pagination, 50 rows per page.
- No N+1 — use select_related / prefetch_related for FK joins.
- Service worker cache bump after CSS/JS changes.


## COMPATIBILITY WITH MASTERMIND ENGINE
=======================================

Quizadmin reads mastermind models directly (read-only in views.py).
Quizadmin calls mastermind.ai_generator functions for mutations
(approve_question, reject_question, bulk_approve_questions).

Zero new tables. Zero new Django models. Zero new migrations.
If we need a per-reviewer metric (e.g. reviews per day), that lives in
mastermind, not here — mastermind is the brain.

## KNOWN RISKS / OPEN QUESTIONS
===============================

R1. pending_review_count context processor runs a COUNT query on every
    page. If the review queue grows past 10K rows, this could slow pages.
    Mitigation: add 30-second Django cache; move to context processor
    AFTER verifying count stays small in real data.

R2. "Next question" chaining depends on stable ordering. If reviewer
    approves question A, then the next page needs a consistent
    ordering function. Use `.order_by('created_at', 'mastermind_coll_
    question_id')` everywhere; document the rule.

R3. Bulk operations could delete/reject many questions at once. Add a
    confirmation step (inline bar, not modal) and an undo window via
    soft delete (is_active flag already exists on CollQuestion).
    Deferred: this pass uses per-row actions only.

R4. If two reviewers open the same question in different tabs, both could
    approve it. Reject should be idempotent (already true in engine).
    Approve should be idempotent too. Verify in implementation.

R5. Dashboard metric queries hit the DB. If dashboard becomes popular,
    cache each metric with a short TTL. In this pass, no cache — we'll
    measure real usage first.
