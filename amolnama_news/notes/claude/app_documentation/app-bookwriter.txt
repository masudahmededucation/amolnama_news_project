  2026-04-27  Single canonical public reading surface — the 3D
              leather-bound book reader. Per-chapter public reader
              URL becomes a redirect.

              **Why**
              Previously two reading experiences for the same content:
                * /bookwriter/read/<id>/         — owner-only 3D reader
                * /bookwriter/read/<slug>/       — public per-chapter reader
              Two surfaces = two implementations = drift over time.
              Consolidated to ONE: the 3D reader is now public-facing,
              SEO-indexed, and the only entry-point every other page
              links to.

              **URL contract**
                * /bookwriter/read/<int:book_id>/<slug:book_name_slug>/
                  — canonical, primary, public, anonymous-friendly.
                * /bookwriter/read/<int:book_id>/  — bare-id route
                  resolved by the same view, which 301-redirects to
                  the slugged URL (one canonical address per book).
                * /bookwriter/read/<slug:public_chapter_slug>/  —
                  legacy per-chapter URL still resolves so previously-
                  shared chapter links don't 404; view 301-redirects
                  to the parent book's 3D reader URL.

              **Anonymous-friendly**
              bookwriter_book_reader no longer redirects anon → library
              and no longer 404s non-owner book ids. Owner-only UI
              (Edit pencil; Close-to-library) is gated behind
              `is_viewer_owner`; non-owners get Close-to-marketplace.

              **Single source of truth (helpers)**
              Added in views_api_helpers.py:
                build_book_reader_canonical_slug(book) → slug
                build_book_reader_canonical_path(book) → path
              build_book_card_payload now emits
              `book_reader_canonical_url` so every card consumer
              (library, marketplace, promo) renders the same href.

              **SEO**
              The 3D reader page now passes a real `seo` context dict:
                * title:        '<book title> · কলম | আমলনামা নিউজ'
                * description:  'Read <book> by <author> on কলম.'
                * canonical:    request.build_absolute_uri()
                * og_type:      'book'
                * json_ld:      schema.org/Book payload
                                (name, author, numberOfPages,
                                 inLanguage, url)
                * noindex:      True only when the book has zero
                                published chapters (don't index
                                draft-only books).

  2026-04-24  Book reader page-face containment + client paginator fixes.

              Two regressions cleared in the 3D book reader
              (/bookwriter/read/<id>/):

              1. Chapter prose was visible BELOW the book area, with
                 the bottom jump-tab pill overlapping it. Root cause
                 was a stale collected static file: source CSS had
                 `overflow: hidden` on
                   .bookwriter-book-reader-page-face
                 (it had been temporarily removed earlier as part of
                 a failed Chrome "snapshot before flip" experiment,
                 then restored), but project_static_collected/ kept
                 the broken version because no collectstatic was run
                 after the restore. Browser + service-worker cache
                 then served the broken file.
                 Fix: ran collectstatic, verified source/collected
                 in sync, bumped CACHE_NAME (v895 → v896) so the
                 service worker re-fetches.

              2. Image inside chapter body could be silently lost or
                 duplicated by the client-side paginator. Root cause:
                 the sentence-split branch of
                   _paginateGhostChildrenIntoPages
                 in modules/bookwriter-reader-client-side-paginator.js
                 appended a working clone paragraph alongside the
                 still-attached original full paragraph in the ghost
                 measurer. Every measurement immediately overflowed,
                 the function pushed the full original as one page
                 AND the same content again as sentence-split pages,
                 which inflated rendered sheet content and could
                 collide with the page-face containment bug above.
                 Fix: explicit
                   ghostElement.removeChild(nodeBeingMeasured)
                 before appending the working clone.
                 Also added Bengali sentence-end punctuation `।`
                 (U+0964) to the sentence-split regex so Bengali
                 prose paginates at natural sentence boundaries.

              Defaulted ENABLE_CLIENT_SIDE_PAGINATION to FALSE in
              pages/page-book-reader.js — server-side word-count
              pagination is the proven, shipping path; the client
              measurement-based paginator is kept in tree but
              disabled until the user opts back in.

              Files touched:

                                            on .page-face)

A standalone writing app inside amolnama_news. Lets a user draft a
book, organize a corkboard of plot cards, build a worldbuilding
"bible", design a cover, share early drafts with beta readers, and
publish chapters one at a time as a serial release. Public reader
page (`/bookwriter/read/<slug>/`) is anonymous-friendly and
SEO-indexable; beta-only reader (`/bookwriter/beta/<token>/`) is
noindex.

App location: amolnama_news/site_apps/bookwriter/
URL prefix:   /bookwriter/
Schema:       [bookwriter]
Brand:        কলম (Bengali for "pen")
Inspiration:  Inkwell design template (paper + ink + Fraunces +
              EB Garamond + JetBrains Mono — "writing sanctuary")

============================================================
  END-TO-END DATA FLOW
============================================================

  WRITER FLOW
  ─────────────────────────────────────────────────────────
  1. Anonymous visitor lands on /bookwriter/ → sees demo content.
  2. Logged-in visitor → _ensure_default_book_for_user creates a
     CollBook + Chapter on first visit (idempotent).
  3. Writer types in editor (.editor[contenteditable]) →
     debounced 800ms autosave to /api/chapter/<id>/autosave/.
     This bumps:
       - chapter.chapter_text_html / chapter_text_plain / word_count
       - book_word_count_cached / book_chapter_count_cached
       - WritingSession (today's session row)
       - EngUserStreak (today's streak row)
  4. Writer flips to corkboard → plot cards CRUD via
     /api/book/<id>/plot-card/create/ + /api/plot-card/<id>/save/.
  5. Writer flips to bible → bible entries CRUD by category.
  6. Writer flips to cover → tile clicks + sliders persist via
     /api/book/<id>/cover-design/save/. State reflected on reload.
  7. Writer publishes chapter → /api/chapter/<id>/publish/ creates
     a serial_release with public_chapter_slug from chapter title
     (via bangla_slugify). Idempotent — re-publishing is safe.

  PUBLIC READER FLOW
  ─────────────────────────────────────────────────────────
  1. Anyone visits /bookwriter/read/<slug>/ → renders the chapter
     prose, increments read_count_cached.
  2. Engagement strip:
       - Subscribe button (logged-in only) → toggles
         engagement_serial_subscriber row.
       - 5 reaction buttons → toggle engagement_serial_reaction
         row (UNIQUE on release+user+kind).
       - Comment thread → engagement_serial_comment with optional
         parent_id for replies. Owner can pin; author or owner
         can delete.
  3. View tracking → POST to /api/release/<id>/view/ on mount.
     Anonymous reads use session_hash; logged-in reads carry
     user_profile_id. Bumps read_count_cached + unique_reader_count.

  BETA READER FLOW
  ─────────────────────────────────────────────────────────
  1. Writer mints a share link via /api/book/<id>/beta-share/create/
     and copies the URL.
  2. Reader visits /bookwriter/beta/<token>/ → auto-creates a
     beta_reader row (linking the visitor to the share link).
  3. Sidebar shows beta comments. Reader can post (kind = comment /
     suggestion / praise) anchored to a passage.
  4. Owner can mark resolved or accept/reject suggestions.

============================================================
  SECURITY GUARANTEES
============================================================

  - All write endpoints: @login_required + @require_POST/_DELETE.
  - Owner check: every endpoint resolves chapter→book→user_profile_id
    via the _resolve_*_for_owner helpers. Cross-owner returns 403.
  - HTML sanitization: chapter body passes through
    sanitize_user_html() (core/utils). Bible / plot / margin notes
    use _strip_html_to_plain (text-only fields).
  - Hex-color injection: every hex color (cover palette overrides,
    bible avatar gradient hexes) validated via _is_valid_hex_color
    against the pattern `^#[0-9a-fA-F]{3,8}$`. Prevents
    style-attribute breakout XSS.
  - Beta token: 32-char URL-safe random via secrets.token_urlsafe.
  - Public reader: anonymous can read + comment list (require_POST
    gates state changes; only logged-in users can subscribe / react /
    post comments).
  - SQL: pure ORM. No raw SQL anywhere.

============================================================
  FILES TO MODIFY WHEN ADDING A NEW FEATURE
============================================================

  Adding a new field that already exists in SQL:
    1. models.py — add the field declaration if not present
    2. views_api.py — extend the relevant save endpoint to accept it
    3. views.py — load it into context if templates need it
    4. _inkwell_body.html — add the form element
    5. page-inkwell.css — style it (use existing tokens)
    6. page-inkwell.js — wire change/input handler

  Adding a new ref group:
    1. SQL: INSERT into ref_<name>
    2. models.py — Ref<Name> class (managed=False)
    3. views_api_refs.py — one entry in REF_GROUP_TABLE_MAP
    4. (Optional) JS hits /api/refs/<group>/ to populate dropdown

  Adding a new endpoint:
    1. views_api.py — add @login_required + @require_POST + resolver
    2. urls.py — register route under api/<resource>/<action>/
    3. JS — call via window.bookwriter.apiPost(url, payload).then().catch()
       — NEVER re-implement fetch+CSRF locally; namespace.js is the
       single source of truth. Use apiDelete / apiGet for the other
       verbs.
    4. For two-click delete affordances, use
       window.bookwriter.wireTwoClickConfirmDelete(button, {
         confirmingClass: 'bookwriter-...',  // class defined in CSS
         initialTitle: '...', confirmingTitle: '...',  // optional a11y
         onConfirm: function () { window.bookwriter.apiDelete(...); }
       });
    5. Verify via Django shell with RequestFactory before claiming done

  Adding a new CSS class:
    - ALWAYS prefix with `bookwriter-`. No exceptions.
    - Touch-target fallback values must be 44px to match
      `--touch-target` defined globally in core/colors.css. Pattern:
      `min-height: var(--touch-target, 44px);`
    - Mobile coverage already provided by the `@media (pointer: coarse),
      (max-width: 760px)` block in page-inkwell.css which forces 44px
      min-height on every button/input/select/textarea inside
      .bookwriter-shell / .bookwriter-embedded-wrapper.

  Dynamic per-row inline styles (CSS-variable injection pattern):
    - When a value MUST come from the database (per-row hex pair,
      per-record dimensions), do NOT emit a full design rule inline.
    - Declare the design in CSS with custom properties + sensible
      fallbacks, then inject ONLY the data:
        CSS:
          .bookwriter-bible-avatar {
            --bookwriter-bible-avatar-hex-1: var(--ochre);
            --bookwriter-bible-avatar-hex-2: var(--accent);
            background: linear-gradient(
              135deg,
              var(--bookwriter-bible-avatar-hex-1),
              var(--bookwriter-bible-avatar-hex-2));
          }
        Template:
          <div class="bookwriter-bible-avatar"
               {% if entry.hex_1 and entry.hex_2 %}
                 style="--bookwriter-bible-avatar-hex-1:{{ entry.hex_1 }};
                        --bookwriter-bible-avatar-hex-2:{{ entry.hex_2 }};"
               {% endif %}>
        JS (when creating dynamically):
          el.style.setProperty('--bookwriter-bible-avatar-hex-1', hex1);
          el.style.setProperty('--bookwriter-bible-avatar-hex-2', hex2);
    - Why: design (135deg, fallback colours, shape) lives once in CSS
      where future tweaks are easy. Inline carries strictly DATA, never
      DESIGN. Reference sites: bible avatar / bible portrait at
      _inkwell_body.html:882 and :924, page-inkwell.js:1113/1523/1803.

  Cover palette tiles also use this pattern:
    - Each tile carries a palette-name class
      (`.bookwriter-cover-palette-tile-classical` etc.) that sets
      `--bookwriter-palette-{bg,fg,accent}` custom properties.
    - The 3 swatch spans inside read those variables — no inline style.
    - Adding a new palette = one new class block in CSS + one tile in
      the template (no inline style needed).

  Inline-style audit (bookwriter is currently 100% compliant):
    - Render inkwell + grep rendered HTML for `style="`.
    - Acceptable: 0 in inkwell mode, only conditional bible-avatar inline
      attrs when DB rows have hex pairs set (carries only --hex-1/--hex-2
      custom properties — data injection only).
