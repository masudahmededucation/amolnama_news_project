================================================================================
PHASE 3 — blog structural alignment (Option X, additive)
================================================================================

SQL script: notes/sql-script/phase3-blog-structural-alignment.sql

Goal: make every blog collection table share the same universal fields
(is_published, is_active, is_featured, bookmark/like/view/comment counts,
summary_bn/en) so shared code doesn't need to know which fields exist where.

stories/coll_story was the gold standard — already had everything.

Changes per table
-----------------
art/coll_artwork:
  + artwork_summary_bn NVARCHAR(500) NULL
  + artwork_summary_en NVARCHAR(500) NULL
(Already had all counters + flags.)

poem/coll_poem_entry:
  + is_active         BIT NOT NULL DEFAULT 1
  + is_featured       BIT NOT NULL DEFAULT 0
  + bookmark_count    INT NOT NULL DEFAULT 0
  + comment_count     INT NOT NULL DEFAULT 0
  + poem_summary_bn   NVARCHAR(500) NULL
  + poem_summary_en   NVARCHAR(500) NULL
  + is_published      AS (CASE WHEN poem_status_code = N'published'
                              THEN 1 ELSE 0 END) PERSISTED

bangladesh/coll_destination:
  + is_active         BIT NOT NULL DEFAULT 1
  + comment_count     INT NOT NULL DEFAULT 0
  + is_published      AS (CASE WHEN destination_status = N'published'
                              THEN 1 ELSE 0 END) PERSISTED

bangladesh/coll_media_entry:
  + is_active         BIT NOT NULL DEFAULT 1
  + is_featured       BIT NOT NULL DEFAULT 0
  + bookmark_count    INT NOT NULL DEFAULT 0
  + media_slug        NVARCHAR(400) NULL
  + is_published      AS (CASE WHEN media_status = N'published'
                              THEN 1 ELSE 0 END) PERSISTED

debate/coll_topic:
  + bookmark_count    INT NOT NULL DEFAULT 0
  + like_count        INT NOT NULL DEFAULT 0
  + view_count        INT NOT NULL DEFAULT 0
  + comment_count     INT NOT NULL DEFAULT 0

Why PERSISTED computed is_published
------------------------------------
Stories + art store is_published as a real boolean column. Poem, destination,
media use string status codes (poem_status_code, destination_status,
media_status). Adding a parallel is_published BIT column would create two
sources of truth that drift. A PERSISTED computed column stays canonically
in sync automatically — SQL Server recomputes on UPDATE, no triggers needed.

Python side: Django models cannot declare a computed column as a field
(ORM would include it in INSERT/UPDATE, SQL Server rejects). Instead,
poem/destination/media models.py exposes is_published as a Python @property
that reads status_code. Shared code calls row.is_published uniformly,
writes never touch it. Same answer from both sides.

Why no is_published on debate
------------------------------
Debate has a richer lifecycle — scheduled / live / paused / closed /
archived / winner_decided — than a boolean can represent. The existing
link_blog_debate_ref_topic_status_id FK models this correctly. Debate
still gets the 4 universal counters so cross-content features (bookmark,
like, view, comment) work.

Database cross-reference — extended properties
-----------------------------------------------
The SQL script is auto-generated from the Python registry and stamps
'registry_role' extended properties on every registered column. After
running it, SSMS tooltips show the role when hovering a column, and
this query returns the full registry in tabular form:

    SELECT OBJECT_SCHEMA_NAME(ep.major_id) AS schema_name,
           OBJECT_NAME(ep.major_id)        AS table_name,
           c.name                           AS column_name,
           CAST(ep.value AS NVARCHAR(500)) AS registry_role
    FROM sys.extended_properties ep
    JOIN sys.columns c
         ON c.object_id = ep.major_id AND c.column_id = ep.minor_id
    WHERE ep.name = N'registry_role'
    ORDER BY schema_name, table_name, column_name;

Idempotent — DROP + ADD per column — so after any edit to the Python
registry, re-run the script and the DB cross-reference updates in place.

================================================================================
LESSONS LEARNED (write to CLAUDE.md troubleshooting if recurring)
================================================================================

1. SQL Server computed columns: never declare them as Django model fields.
   Django ORM includes them in INSERT/UPDATE; SQL Server rejects. Use a
   Python @property that reads the base column, and keep the DB PERSISTED
   computed as the canonical answer.

2. ref_content_subcategory has TWO computed columns (link_subcategory_id,
   link_subcategory_code) that INSERT statements must NOT name. Always list
   columns explicitly — never use SELECT *.

3. Silent field-name typos across JS <-> API: grep-verify both sides of
   every POST key rename. Three JS/API mismatches (destination category,
   best season, season) were silently killing create flows until Phase 2
   surfaced them.

4. Additive migrations (Option X) are the professional default: no
   renames on existing columns, all new columns NULL or DEFAULTed,
   everything wrapped in a single SQL transaction with TRY/CATCH +
   XACT_ABORT ON. Idempotent guards (IF EXISTS / COL_LENGTH) make
   scripts re-runnable after partial success.

5. Extended properties are a gold-standard way to document DB-side roles
   without having to manually keep docs in sync. Generate them from the
   Python source of truth, not the other way around.
