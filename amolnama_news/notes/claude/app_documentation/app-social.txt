SOCIAL APP — Architecture & Features
App: site_apps/social/
URL prefix: /social/
================================================================

## Overview
User-to-user follow/unfollow system. Lightweight — just one model and one API.

## Models

| Model       | DB Table                | Purpose                     |
|-------------|-------------------------|-----------------------------|
| UserFollow  | [social].[user_follow]  | Follower/following tracking |

### Key Fields on UserFollow
- user_follow_id (PK)
- link_follower_user_profile_id (FK → user_profile, the follower)
- link_following_user_profile_id (FK → user_profile, being followed)
- is_active (soft toggle — re-activate on re-follow, deactivate on unfollow)
- created_at

## URL Patterns

| URL                              | View               | Method | Name                     |
|----------------------------------|---------------------|--------|--------------------------|
| /social/                         | home                | GET    | social:home              |
| /social/api/follow/<id>/         | api_follow_toggle   | POST   | social:api_follow_toggle |

## API: api_follow_toggle

POST /social/api/follow/<user_profile_id>/
- Self-follow blocked (returns error)
- 3-way toggle: create new / re-activate / deactivate
- Follower count recalculated from actual data (not increment/decrement)
- Returns: { success, following (bool), follower_count }

## Integration Points
- post-feed-interactions.js calls this API via Follow buttons on post cards
- Profile hover popup shows follower/following counts + follow button
- Feed "Following" tab filters posts by followed user IDs
- All follow buttons for same user update simultaneously (querySelectorAll by data-user-profile-id)

## Features

### Follow/Unfollow Users
- Click "Follow" button on any post card → follows the post author
- Button changes to "Following" (checkmark icon, filled blue)
- Click again → unfollows (reverts to "Follow" with person+ icon)
- All instances of the same user's follow button update simultaneously across the feed

### Profile Hover Popup
- Hover over author name → popup shows avatar, @handle, bio, follower/following counts
- Follow button inside popup works identically to inline button

### Following Feed Tab
- "Following" tab on homepage shows only posts from followed users
- Empty state: "আপনি এখনো কাউকে অনুসরণ করেননি" with count display

### Follower/Following Counts
- Counts recalculated from actual DB data (COUNT WHERE is_active=True)
- Displayed in profile popup and portal profile page

## Functions

### api_follow_toggle(request, user_profile_id) — views_api.py
Handles follow/unfollow toggle with 3-way pattern:
1. No existing record → creates new UserFollow (is_active=True)
2. Existing active record → deactivates (is_active=False) = unfollow
3. Existing inactive record → re-activates (is_active=True) = re-follow
Self-follow blocked with error response. Returns actual follower count.

### home(request) — views.py
Placeholder landing page. Redirects or renders base template.

## Static Files
None — all UI is in post-card.html template and post-feed-interactions.js.

## Troubleshooting

### Follow button not updating all instances
All follow buttons for the same user use `data-user-profile-id` attribute.
The JS handler queries ALL buttons with matching attribute and updates them.
If a new post card is added dynamically (e.g., after repost), its follow button
may not be in sync — page reload resolves this.

### Self-follow appearing
The template hides follow buttons when `can_edit` is true (own posts).
For reposts, `original_is_same_user` flag hides follow on self-reposts.
If follow button appears on own content, check these template conditions.

### Count drift
Follower counts are always recalculated from actual data (not increment/decrement).
If counts seem wrong, check for orphan records where is_active logic may have been bypassed.

## Phase 3b Refactor: public_profile_articles

Commit 569110d (2026-04-11) replaced an if/elif chain with a generic
spec-driven lookup using content_type_registry.

### Old shape
```python
from amolnama_news.site_apps.art.models import CollArtwork
from amolnama_news.site_apps.stories.models import CollStory
from amolnama_news.site_apps.bangladesh.models import CollDestination
from amolnama_news.site_apps.poem.models import CollPoemEntry

for item in registry_items:
    type_code = ...
    slug = item.content_slug or ''
    if type_code == 'art' and slug:
        row = CollArtwork.objects.filter(artwork_slug=slug).only('...').first()
        source_id = row.blog_art_coll_artwork_id if row else None
    elif type_code == 'story' and slug:
        row = CollStory.objects.filter(story_slug=slug).only('...').first()
        source_id = row.blog_stories_coll_story_id if row else None
    elif type_code == 'destination' and slug:
        ...
    elif type_code == 'poem' and slug:
        ...
```

### New shape
```python
from amolnama_news.site_apps.content.content_type_registry import (
    get_spec, get_model_class,
)

for item in registry_items:
    type_code = ...
    slug = item.content_slug or ''
    if not type_code or not slug:
        continue
    spec = get_spec(type_code)
    if not spec or not spec.get('slug_field') or not spec.get('pk_field'):
        continue
    model_class = get_model_class(type_code)
    if model_class is None:
        continue
    row = (model_class.objects
           .filter(**{spec['slug_field']: slug})
           .only(spec['pk_field'])
           .first())
    if row is not None:
        source_id = getattr(row, spec['pk_field'], None)
        if source_id:
            registry_to_source_id[item.content_registry_id] = (type_code, source_id)
```

No hardcoded model imports. Adding a new blog content type to the
articles profile page now requires zero edits — just add a spec to
content_type_registry.

Covers still resolved via `get_cover_urls_for_content_refs()` (bulk
lookup per content type). Presentation metadata (badge, color, cta)
still resolved via `get_content_type_metadata()` — both of those
helpers now read from the registry too.

## New Module: views_bookmarks.py (2026-04-10)

Previously bookmarks was part of views.py. Split into its own file as

## Dual-Active Sidebar Nav Fix (commit 8d131e6, 2026-04-11)

Social was the most visible place this bug surfaced. When a user was
on `/social/bookmarks/`, BOTH the Social nav item AND the Bookmarks nav
item lit up. F5 fixed it temporarily; next SPA click brought it back.

### Root cause
The client-side SPA navigation in core/spa-navigation.js used
`url.startsWith(href)` to decide which sidebar item was active. That
gave a false positive for `/social/bookmarks/` matching both `/social/`
AND `/social/bookmarks/`.

Earlier session had already fixed the server-side context processor
(`core/context_processors.py:active_sidebar_nav` + ordered
SIDEBAR_NAV_RULES that match exact paths before namespace fallbacks),
but the JS rebuilt the same bug after the content swap.

### Fix
Server already computes the correct single nav id. Embed it on the
page itself so the JS can read it instead of re-deriving:

1. `core/templates/core/base.html` —
   `<main data-active-sidebar-nav-id="{{ active_sidebar_nav_id }}">`
2. `spa-navigation.js:updateSidebarActiveState()` now reads that data
   attribute from the parsed `<main>` after content swap and toggles
   sidebar items by matching the suffix of their id
   (`sidebar-navigation-item-X`). Single source of truth on both
   server AND client — structurally impossible to dual-light.

Verified end-to-end via authenticated test client:
```
/                          -> home       (only)
/social/bookmarks/          -> bookmarks  (only)
/social/@joes/articles/     -> social     (only)
```

No social-app-specific changes — the fix lives in core. Mentioning it
here because social was the reproduction case.

## UserBlock Integration

social.UserBlock is the bidirectional block model. Checked by:
- `messenger.views_api._is_blocked()` — rejects new conversations /
  new messages between a blocked pair
- `post/views.py` feed queries — filters out posts from blocked users
- `social.views.api_block_toggle` — toggles the block row

The check is bidirectional: if A blocks B, neither can message the
other or see the other's content.
