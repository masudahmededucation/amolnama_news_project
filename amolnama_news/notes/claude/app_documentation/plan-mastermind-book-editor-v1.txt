Last updated: 2026-04-17.


SCOPE — 4 features unified into ONE editor + 1 reject-recovery flow
-------------------------------------------------------------------
1. Book reader / editor for scanned (imported) books — staff can read OCR'd
   chapters, fix OCR errors, add chapter#/name, save back.
2. Paste-as-book ingest — skip OCR entirely; paste plain text → auto-chunk
   into chapters → save.
3. Rejected-question review-and-edit — currently rejected = soft-deleted +
   archived (unrecoverable). Add "Needs edit" status that lets reviewers
   recover near-misses, fix them in the existing editor, and approve.
4. Book writing tools — any author can write + edit + publish original
   books on the platform. Same editor as #1 + #2; new origin code.

Design insight: features #1, #2, #4 are the same primitive (book editor)
with 3 doorways (imported / pasted / authored). Build ONE editor, three
entry points. Reuses the existing CollBook + CollBookChapter + CollBookChunk
data model with minimal additions.


GROUND TRUTH (verified by Explore agent, 2026-04-17)
----------------------------------------------------
- CollBook has 20 columns; missing: link_created_by_user_profile_id,
  book_origin_code, book_status_code, book_published_at, book_slug.
- CollBookChapter has 10 columns (metadata only — no text body).
- Chapter TEXT lives in CollBookChunk (one row per ~500-word paragraph,
  with chunk_text + chunk_page_number + chunk_paragraph_index +
  chunk_sequence_order). Persisted; never discarded.
- AI gen pipeline (ai_generator.py) reads from CollBookChunk.chunk_text.
  Re-chunking after edit means re-running chunker + replacing rows for
  the affected chapter.
- Question reject (reject_question) currently sets question_status_code='archived'
  AND is_active=False — hard-disable, no recovery path.
- review_queue UI shows one question with Approve / Reject. NO in-queue edit.
- question_form.html is a full bidirectional editor that accepts any status
  on save — directly reusable for "needs_edit" recovery flow.
- textextractor has a plaintext engine that takes a FILE PATH, not a string —
  paste-as-book needs a new helper (chunk_plain_text_string) in mastermind.
- CollBook does NOT participate in content/ContentRegistry. For v1 we
  surface the public reader at /mastermind/book/<id>/ without registry
  integration.

NEW STATUS CODE for questions: 'needs_edit'
   No SQL change — question_status_code is NVARCHAR(20) and accepts any
   string. Engine + UI just need to recognise it as a 3rd state alongside
   'review' / 'archived'.

NO new tables needed. CollBook + CollBookChapter + CollBookChunk already
cover all four features.

FLOW DIAGRAMS
-------------
A) Reject-recovery flow:
   AI generates question (status='review')
     → Reviewer opens review queue
     → 3 actions:
         (1) Approve  → status='published', source='ai_reviewed', is_active=True
         (2) Needs Edit → status='needs_edit', is_active=True (NOT archived)
                          → Question appears in /quizadmin/needs-edit/
                          → Staff opens editor → fixes → saves with status='published'
         (3) Reject (garbage) → status='archived', is_active=False (current behavior)
   "Recover" button on archived questions → status='needs_edit' (one-shot recovery
   for accidentally rejected items).

B) Paste-as-book flow:
   /quizadmin/book/create/  (3-card chooser)
     → user clicks "Paste text"
     → /quizadmin/book/create/paste/ (form: title, lang, paste_text)
     → POST /mastermind/api/book/create-from-paste/
        Server: chunk_pasted_text() → CollBookChunk rows + 1 CollBookChapter row
                book_origin_code='pasted_text', book_status_code='draft'
     → Redirect to /quizadmin/book/<id>/edit/

C) Author-write flow:
   /quizadmin/book/create/
     → user clicks "Write a book"
     → POST /mastermind/api/book/create/ with origin_code='user_authored'
     → Redirect to /quizadmin/book/<id>/edit/
     → User adds chapters, types text into each chapter editor
     → Auto-save every 3 seconds OR on blur

D) Editor (shared by all 3 origins):
   Sidebar: chapter list + "+ New chapter" + drag-to-reorder
   Main: active chapter
     - chapter_number (auto, editable)
     - chapter_title_bn / _en
     - chapter_text textarea (large, monospace acceptable for OCR fix flow)
     - "Save" button + auto-save chip
     - "Generate questions from this chapter" button (calls existing AI gen)
     - "Read mode" toggle to preview as paginated reader

E) Public reader:
   /mastermind/book/<id>/   — for status='published' books only; 404 otherwise
   Shows:
     - Book metadata header (title, author, edition, language)
     - Chapter sidebar (sticky on desktop, drawer on mobile)
     - Main reader pane: chapter title + paginated text
     - "Take quiz from this book" CTA if any quiz cites this book

PRE-IMPLEMENTATION BUG CHECK ON THE PLAN
----------------------------------------
✅ DB script idempotent (column-exists guards)
✅ Status code 'needs_edit' is just a string — no DB constraint to update
✅ Permission filter applied at every list endpoint (owner-or-staff)
✅ Public reader 404s on unpublished books — no leak
✅ Re-chunking on save replaces (DELETE + INSERT) per chapter only — won't
   wipe other chapters' chunks
✅ AI gen path unchanged — still reads CollBookChunk.chunk_text
✅ Existing question editor accepts any status — no schema work for the
   needs_edit recovery flow
✅ Paste-text chunker uses paragraph splitting + ~500-word cap (matches
   the existing PDF chunker used by ai_generator.py)
✅ Mobile UX: sidebar collapses to drawer at ≤768px
✅ XSS: book chapter text rendered through Django auto-escape; reader
   converts \n\n to <p> blocks via splitlines + escaping (no |safe)
✅ CSRF: every mutating endpoint uses @require_POST + login_required
✅ All new HTML form elements get id + name attributes
✅ All new CSS uses tokens; @media (prefers-reduced-motion: reduce) included
✅ Auto-save uses throttling (3s) — no excessive server requests
✅ No internal <style> blocks; no inline style except JS-target dynamic values

POTENTIAL RISKS LOGGED (acceptable for v1)
- Concurrent edits: last-write wins. Document as known limitation.
- Very large pastes (>200k words): may take a few seconds to chunk +
  insert. Acceptable for v1; revisit if real users complain.
- Author abandonment: half-written drafts stay forever. Acceptable —
  adds nothing to risk surface.

OUT OF SCOPE for v1 (deferred to v2 if requested)
- Public book browse / search / discovery feed
- Likes / comments / bookmarks on books (use existing universal social
  endpoint when wanted)
- Rich text formatting (Quill/CKEditor) — start with plain textarea
- Multi-author collaboration with conflict resolution
- Book cover image uploader (use existing cover_image_url URL field for now)
- ContentRegistry integration (publish book to public homepage)
- Server-side WYSIWYG drag-to-reorder (V1 uses up/down arrows for chapter
  reordering — same pattern as ordering question)
- Analytics on book reads / downloads
