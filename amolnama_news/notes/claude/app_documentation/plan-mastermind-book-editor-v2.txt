PLAN — MASTERMIND BOOK EDITOR + READER v2 (real functionality, not stubs)
==========================================================================
Status: AWAITING APPROVAL (2026-04-17)
Last updated: 2026-04-17
Plan author: research-driven from 8 leading apps; v1 stub → v2 functional.

WHY v2 EXISTS
-------------
v1 shipped a chapter sidebar + plain textarea + 3 buttons. User feedback was
direct: "absolute rubbish, no functionality. just a few buttons that even
don't work." Root cause: I built the data plumbing without doing the UX
research first. v2 fixes that — research first, then plan, then build.

DEEP RESEARCH (8 apps, real findings)
-------------------------------------
Editors (authoring):
  Reedsy Studio   — LEFT sidebar = chapter list (split/delete/insert);
                    RIGHT sidebar = Goals & Insights (daily word count,
                    chapter word/char counts, pinned notes, comments);
                    autosave; spellcheck; search-in-chapter; toolbar on
                    the right side. Source: writersanctuary.com,
                    blog.reedsy.com.
  Substack        — Rich-text editor; drafts auto-saved; can only edit
                    one draft at a time; word count visible; SEO metadata
                    panel; post scheduling; version history; cover/audio/
                    video upload. Source: substack.com/features,
                    on.substack.com/p/mobile-editor.
  Wattpad         — Cover-first creation flow: title → description →
                    main characters → category → tags → audience →
                    language → rating BEFORE writing the first chapter.
                    Then per-chapter rich-text edit, scheduling, and
                    end-of-chapter calls-to-action. Source: wattpad.com,
                    selfpublishing.com/wattpad-review.
  Atticus.io      — Browser-based, cross-platform; built-in writing
                    interface with distraction-free mode + word count
                    tracking + chapter organisation + note-keeping;
                    callout boxes, sidebars, footnotes/endnotes for
                    nonfiction. Source: kindlepreneur.com/atticus-vs-vellum.
  Vellum          — NOT a writing app — formatting/design only.
                    Writers do prose elsewhere. Source: same.
  Notion          — Block model; LEFT sidebar = pages tree; live cursor
                    presence; comments; offline saves and syncs;
                    keyboard shortcuts for new page. Source:
                    notion.com/notes, ones.com.

Readers (consumption):
  Kindle web      — 14 font sizes; font family picker (incl. OpenDyslexic);
                    4 page colours (white/sepia/black + custom themes);
                    1-or-2-column layout; narrow/medium/wide margins;
                    line/paragraph/word/character spacing controls;
                    continuous scroll OR page-turn; progress shown as
                    "minutes left in chapter" OR location number.
                    Source: amazon.com customer-service docs.
  Apple Books     — 9 fonts; bookmarks per page; highlights with optional
                    notes; search-inside-book; cross-device sync of
                    bookmarks/highlights/notes; sepia + night-mode + light
                    themes; line/character/word spacing; auto-saves
                    reading place. Source: support.apple.com.

Integrated learning:
  Khan Academy    — Reading + practice on the SAME surface. Khanmigo
                    generates custom quizzes from passages with no limits
                    no extra logins. Mastery: extra shot at proficiency
                    after one wrong answer. Source: blog.khanacademy.org
                    Spring 2025 + 2025-26 updates.

KEY TAKEAWAYS (top 12 features that are tractable for plain Django + plain JS)
-----------------------------------------------------------------------------
For the EDITOR (Quiz Panel side, staff/author):
  1. Inline chapter rename (click title → editable in place) — Notion + Reedsy
  2. Chapter delete + reorder (up/down arrows) — every authoring tool
  3. Live word count + reading-time estimate on active chapter — Reedsy + Atticus
  4. Last-saved timestamp ("Saved 2 min ago") not just "Saved" — Notion + Substack
  5. Book metadata side panel: cover image URL, author bn/en, publisher,
     edition, ISBN, language, description, audience — Wattpad cover-first
     flow + Substack metadata panel
  6. "Generate quiz from this chapter" button on each chapter — OUR differentiator
     (Khan Khanmigo pattern)
  7. Book-level publish flow with confirmation chip — Substack pattern
  8. Distraction-free toggle (hides chrome, expands editor) — Atticus
  9. Mobile chapter drawer (sidebar slides in from left) — every modern editor

For the READER (public side):
  10. Font size +/- (3 sizes is enough; Kindle's 14 is overkill for v2)
  11. Theme switcher: light / sepia / dark — universal
  12. Reading progress bar + "minutes left in chapter" estimate — Kindle pattern

OUT OF SCOPE for v2 (deferred to v3):
  - Rich-text editor (Quill/TipTap) — plain textarea acceptable, all writers
    in our user base will be Bengali; Quill's Bengali support is mediocre
  - Real-time multi-user collaboration (Notion-class) — needs Channels
    upgrade + presence tracking
  - Bookmarks / highlights / notes / annotations on the reader side
  - Cross-device sync of reading position
  - Find & replace inside the editor (browser Ctrl+F suffices on textarea)
  - Spellchecker (browser native handles Bengali poorly anyway)
  - Multi-column reader layout
  - Audio narration (huge scope — separate project)
  - Comments on book chapters (use existing universal comments later)
  - Search inside published book (defer to full-text search v3)

DATA MODEL CHANGES (none — purely Python + UI)
----------------------------------------------
Everything fits in the existing schema:
  - Cover image       → book_cover_image_url (already in CollBook)
  - Author bn/en      → already in CollBook
  - Publisher bn/en   → already in CollBook
  - Edition / ISBN    → already in CollBook
  - Language          → already in CollBook
  - Description       → already in CollBook
  - Owner             → link_created_by_user_profile_id (added in v1)
  - Origin            → book_origin_code (added in v1)
  - Status            → book_status_code (added in v1)
  - Slug              → book_slug (added in v1)
  - Chapters          → CollBookChapter (already exists)
  - Chapter text      → CollBookChunk.chunk_text (already exists)
  - Word count        → derived from chunk_word_count sum (no storage)
  - Reading-time est. → derived from word count / 200 wpm

NO SQL CHANGES.
NO new tables.
NO new columns.

STYLING (gen-z, mobile-first, professional, tokenised, accessible)
------------------------------------------------------------------
Editor:
  - 3-column desktop layout via CSS grid
  - Tablet (≤1100px): metadata panel folds into a <details> accordion
    above the editor
  - Mobile (≤768px): chapter sidebar becomes a fixed drawer (slide-in
    from left); body gets `aria-hidden="true"` while drawer is open
  - Word-count chip + last-saved chip in the editor header
    (live-updating, color-coded: idle/typing/saving/saved/error)
  - Distraction-free toggle button → adds .quizadmin-book-editor--focus
    class to the root, hides sidebar + metadata panel
  - All transitions wrapped in @media (prefers-reduced-motion: reduce)
  - All buttons get focus-visible 2px accent outline + min-height
    var(--touch-target)

Reader:
  - Reader controls bar sticky at top of viewport
  - Font size: small=var(--text-base), medium=var(--text-lg),
    large=var(--text-xl); applied via [data-reader-font-size="..."]
    attribute on the reader root
  - Theme: light (default), sepia (#f4ecd8 / #5b4636 text),
    dark (#1a1a1a / #e6e6e6 text); applied via [data-reader-theme="..."]
    on root
  - Reading progress bar: 4px gradient fill (var(--accent) →
    var(--accent-strong)) at the top of the reader pane, updated as
    chapter changes
  - Sticky chapter header: chapter title fades in as the user scrolls
    past the original title element
  - Prev/next chapter buttons in footer; full keyboard support
    (← → arrows)
  - Theme + font-size persisted in localStorage (reader_theme,
    reader_font_size)

PRE-EXECUTION BUG-CHECK ON THE PLAN ITSELF
------------------------------------------
✅ No SQL changes (zero schema risk)
✅ All new endpoints permission-gated (owner+staff via existing helper)
✅ chapter_word_count + book_word_count are pure SUM queries on
   CollBookChunk.chunk_word_count — no N+1 risk
✅ delete_chapter is soft-delete (is_active=False) — recoverable
   if user changes mind
✅ reorder_chapter swaps sort_order between adjacent chapters; no
   re-numbering of all rows (O(1) update, not O(n))
✅ Word-count chip updates client-side on `input` event — no server
   round-trip, accurate enough (split on whitespace)
✅ Last-saved timestamp uses Date.now() at save time, displayed as
   "X seconds/minutes ago" — refreshed every 15s via setInterval
✅ Distraction-free mode is a CSS class toggle on root — zero JS state
   beyond a boolean, no API
✅ Mobile drawer uses a backdrop overlay; click-outside closes;
   ESC key closes; aria-hidden management
✅ Reader theme uses CSS attribute selector ([data-reader-theme="dark"]),
   NOT class swap — easier to override per-page
✅ Reader theme + font-size persisted in localStorage (try/catch wrapped
   for private mode)
✅ Reader prev/next chapter buttons handle the boundary cases (first/last
   chapter)
✅ Keyboard shortcuts skip when activeElement is INPUT/TEXTAREA/SELECT
✅ All new HTML form elements get id + name
✅ All new CSS uses var(--*) tokens; @media reduced-motion guard;
   focus-visible on every interactive
✅ AI-gen button calls existing /mastermind/api/book/<id>/generate/
   endpoint — no new infrastructure
✅ NO use of console.log/debug/info/warn; NO use of alert/confirm/prompt
✅ NO inline event handlers in HTML
✅ NO internal <style> blocks; NO inline style="" except dynamic
   JS-target widths
✅ All raw SQL prevented — pure Django ORM (CollBookChunk
   .aggregate(Sum) for word counts)
✅ No XSS — all user-controlled text rendered via Django auto-escape OR
   _escapeHtml in JS

POTENTIAL RISKS (acceptable for v2)
- Concurrent edits across staff users → last-write wins (same as v1)
- Very long chapters (>10k words) → textarea may feel sluggish on
  low-end mobile; revisit if a writer complains
- Reorder up/down repeated 50 times → 50 small API calls (O(1) each);
  acceptable but could be batched in v3 if observed
