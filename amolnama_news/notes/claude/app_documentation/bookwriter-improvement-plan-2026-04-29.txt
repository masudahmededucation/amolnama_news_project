        File: bookwriter-cover-designer.js

          '.bookwriter-cover-template-tile'
        ) etc. every click; nested handler re-queries again.
        Fix: Cache the NodeList at init (once) and reuse in
          every click handler.
        Test: Click each template/palette/background tile;
          previous-tile deselect + new-tile select still works.
        Risk: Very low. Pure refactor, no API change.
        Status: PENDING

[ ] 1.2  Batch chapter prefetch in feedback (~1 hr)
        File: views_api_collaboration.py around lines 440-540
        Issue: Per-comment Chapter.objects.filter(...).first()
          runs once per row in a 500-row loop.
        Fix: Collect all chapter IDs from comment rows, then
          Chapter.objects.in_bulk(chapter_ids) once into a
          dict, look up per row from the dict.
        Test: Open feedback page on a book with multiple
          chapter comments; chapter info still renders per row.
        Risk: Very low. ORM in_bulk is well-tested.
        Status: PENDING

[ ] 1.3  Bulk-validate ref enums (~2 hrs)
        Files: views_api_book.py + views_api_collaboration.py
        Issue: 6 separate RefXxx.objects.filter(...).exists()
          calls per POST.
        Fix: Pre-load ref tables once at request start into
          frozensets; validate in-memory.
        Test: POST each affected endpoint with valid + invalid
          codes; same accept/reject behavior.
        Risk: Low. Single SELECT replaces 6.
        Status: PENDING

[ ] 1.4  Parameterise 6 _resolve_*_for_owner functions (~1 hr)
        File: views_api_helpers.py around lines 266-560
        Issue: 6 near-identical functions
          (_resolve_chapter_for_owner, _resolve_book_for_owner,
          _resolve_plot_card_for_owner, _resolve_bible_entry_for_owner,
          _resolve_margin_note_for_owner, _resolve_beta_reader_for_owner)
          each doing lookup-verify-owner.
        Fix: Single _resolve_model_for_owner(Model, id, user_id,
          [extra_owner_check]) that takes the model class + id.
          Update every caller to pass the model class.
        Test: Each endpoint that previously called the per-model
          resolver still returns same response on owner / non-owner
          / missing-row cases.
        Risk: Low. Mechanical refactor — pass Model class as arg.
        Status: PENDING

[ ] 1.5  Extract @owner_and_json_validated decorator (~1-2 hrs)
        Files: views_api_book.py, views_api_chapter.py,
          views_api_collaboration.py (9 endpoints)
        Issue: Each endpoint repeats ~10 lines of auth + JSON-parse
          + owner-resolve boilerplate.
        Fix: Decorator that handles user_profile_id resolution,
          JSON body parse, owner-resolve (using the parameterised

        Test: Each decorated endpoint still returns 200 for owner
          + valid payload, 403 for non-owner, 400 for invalid JSON.
        Risk: Low. Endpoint shape unchanged; boilerplate moves
          out of function bodies.
        Status: PENDING
