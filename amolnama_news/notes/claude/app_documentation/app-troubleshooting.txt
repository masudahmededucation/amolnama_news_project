### R-bookwriter-book-reader-content-bleeds-below-page-and-overlaps-jump-tab. Chapter prose visible BELOW the book, with the jump-tab pill overlapping it
Date: 2026-04-24
Was: In the 3D book reader (/bookwriter/read/<id>/), once the user
     was a few sheets in, prose from a different chapter was visible
     in the strip BELOW the book area, and the bottom jump-tab
     ("N / M  Page #  Go") was painted over that bleed-through
     content. The jump-tab itself sat ~16px below the book bottom,
     correctly positioned via
       top: calc(40px + var(--bookwriter-reader-book-height) + 16px);
     so the tab wasn't moving — content was escaping the page
     downward and showing past the book's bottom edge.
Root cause: TWO independent things stacked.
     A. CSS: `.bookwriter-book-reader-page-face` had `overflow: hidden`
        REMOVED earlier in the session as a (failed) attempt to fix
        the Chrome "jumpy snapshot before flip" glitch by avoiding the
        3D-grouping flatten pass. The chapter body inside still had
        overflow:hidden, but the FACE itself no longer clipped — so
        any padding overshoot or 3D rendering quirk could let inner
        content paint outside the face's box. The source CSS was
        later restored to include `overflow: hidden;` on .page-face,
        BUT collectstatic was not re-run, so project_static_collected/
        kept serving the broken (no-overflow-hidden) version. Browser
        + service-worker cache then served the broken file.
     B. JS paginator (when ENABLE_CLIENT_SIDE_PAGINATION=true):
        in modules/bookwriter-reader-client-side-paginator.js the
        sentence-split branch of `_paginateGhostChildrenIntoPages`
        appended a working clone paragraph alongside the original
        (still-attached) full paragraph in the ghost. The ghost
        therefore contained BOTH, every measurement immediately
        overflowed, and the function pushed the full original
        paragraph as its own page AND the same content again as
        sentence-split pages — producing duplicated prose that
        increased the per-sheet content size and exacerbated any
        clipping miss.
Fix (three-layer):
     1. CSS: re-confirmed `overflow: hidden;` on
        `.bookwriter-book-reader-page-face` in pages/page-book-reader.css
        (it had already been restored in source — the missing piece
        was deployment).
     2. Deployment: ran `python ../manage.py collectstatic --noinput`
        from the amolnama_news/ dir; verified
          diff -q site_apps/.../page-book-reader.css
                  project_static_collected/.../page-book-reader.css
        returned no output (in sync).
     3. JS paginator: removed the original full-paragraph node from
        the ghost BEFORE appending the working sentence-split clone:
          if (ghostElement.contains(nodeBeingMeasured)) {
            ghostElement.removeChild(nodeBeingMeasured);
          }
     Also bumped CACHE_NAME in seo/views.py (v895 → v896) so the
     browser + service worker re-fetch the corrected static files
     instead of serving cached copies.
     Also defaulted ENABLE_CLIENT_SIDE_PAGINATION to FALSE in
     pages/page-book-reader.js — the server-side word-count
     pagination is the proven path; the client measurement paginator
     is kept in tree but ships disabled until field-tested.
Lesson:
     - Whenever an animation tweak removes a containment property
       (overflow, clip-path, contain), the FIRST regression to check
       is whether content can now escape its box. Snapshot/flicker
       fixes that touch containment must verify content still clips.
     - CSS source/collected drift is a recurring pollution source.
       After ANY CSS edit, run collectstatic in the same edit pass
       and grep-verify the change appears in project_static_collected/
       BEFORE bumping the cache version. CACHE_NAME bump alone does
       not deploy — collectstatic does.
     - Sentence-split pagination must always strip the original
       block before appending the working clone, otherwise content
       is measured twice and emitted twice.

### R-bookwriter-pdf-export-page-break-overlay-text-leak. Visible "page break" + "1, 2, 3" text inside PDF chapter content
Date: 2026-04-24
Was: Book PDF export rendered the visual page-break overlay markup
     (page-number pills "1, 2, 3" and "page break" labels) as visible
     text inside the chapter content. The overlay is a JS-injected UI
     decoration meant to live ONLY in the live editor, never in
     persisted HTML.
Root cause: the autosave in pages/page-inkwell.js read prose.innerHTML
     wholesale, which included the .bookwriter-page-break-overlay div
     subtree (because the overlay is a direct child of the contenteditable
     prose — it has to be, for position:absolute to resolve against the
     prose box). The server sanitize_chapter_prose_html let the divs
     through (global _SANITIZE_ALLOWED_TAGS includes <div>+class). So
     chapter_text_html in DB was contaminated with the overlay markup.
     PDF export read that polluted HTML and rendered it; book_pdf.html
     has no CSS rule to hide .bookwriter-page-break-overlay (it's a
     standalone PDF template, not the editor's CSS). Pre-flight check
     on real data: 8/8 active chapters in book #4 had the pollution.
Fix (two layers, both additive):
     A. JS autosave clone-and-strip in pages/page-inkwell.js:
        getProseInnerHtmlWithoutPageBreakOverlay(proseElement) clones
        the prose, removes .bookwriter-page-break-overlay from the
        clone, returns cleaned innerHTML. Live editor view unchanged
        (clone is throwaway, the original prose still has the overlay
        for visual page markers).
     B. Server-side strip in views_api_helpers.py:
        strip_page_break_overlay_from_html(html_text) uses Python's
        stdlib html.parser with a depth counter so nested <div>s
        inside the overlay are correctly stripped. Fast-path return
        when the marker class is not in the input. Used by
        api_bookwriter_book_export_pdf when building the chapter
        render items, so legacy polluted DB rows get cleaned at
        render time without a DB migration.
     End-to-end verified via Django shell against book #4: PDF 200 /
     37,998 bytes / 10 pages / 0 "page break" leaks / 0 overlay
     class leaks in extracted PDF text.
Lesson: any UI decoration injected INSIDE a contenteditable element
     must be clone-stripped before persistence, AND any server-side
     renderer that consumes the persisted HTML must defensively strip
     too in case legacy data carries the decoration. Single source of
     truth lives in views_api_helpers.py — future consumers (public
     reader, ePub export, etc.) should call the same helper.

### R-bookwriter-empty-chapter-no-cursor-no-typing. Brand-new chapter showed no cursor; typing produced nothing
Date: 2026-04-24
Was: After creating a new chapter (or switching to a chapter whose
     content was empty), the cursor was invisible in the prose page,
     typing produced no visible text, and the Bangla suggestion
     dropdown anchored to {0,0,0,0} caret rect (parking itself on the
     page-number pill in the right margin). Three visible symptoms.
     Root cause: ONE shared precondition. The contenteditable="true"
     prose div ended up with its ONLY child being the
     contenteditable="false" + position:absolute page-break overlay
     (a div introduced in commit 09b9474 on 2026-04-24 by the visual
     A4 page-break feature). With no text-bearing nodes in the layout
     flow (no <p>, no <br>, no text node), Chromium has nowhere in
     the box to draw the caret. Symptoms cascade:
       (a) no cursor (no in-flow text container),
       (b) execCommand('insertText') has no valid Selection target —
           text either gets inserted into the non-editable overlay
           (silently rejected) or into the wrong subtree,
       (c) Range.getBoundingClientRect() returns all zeros → the
           BanglaInput "if caretRect.bottom > 0" guard skipped the
           position update → dropdown stayed at last-shown coords.
     Bug was DORMANT until 2026-04-24 because every chapter the user
     edited until then already had <p> content from past sessions.
     The first new-chapter creation today exposed it.
Fix: ensureProseHasParagraphContainer(proseElement) helper in
     pages/page-inkwell.js. After every prose.innerHTML assignment
     (5 sites), if prose has no <p> direct child, prepend a
     <p><br></p>. Idempotent — skipped when a <p> already exists.
     Inserted at the FRONT so the page-break overlay (always last
     child) keeps its expected position.
     Companion helper focusProseAndPlaceCaretAtFirstParagraph(prose)
     uses Range.selectNodeContents + collapse(true) to put the caret
     at offset 0 of the new <p>, then prose.focus(). Called only when
     the placeholder was actually injected (chapter was empty), so
     existing-content chapter switches don't get an unwanted focus
     jump.
     Two earlier defensive fixes from this session (v861) become
     belt-and-braces:
       - BanglaInput empty-rect fallback in bangla-input.js
       - findLastContentBottomPxRelativeToProse for page count in
         bookwriter-page-breaks.js
     Both still ship; with the placeholder, the caret rect is now
     non-zero on empty chapters, so neither path is the primary fix.
Cache v864 -> v865.
Lesson: a contenteditable element that contains ANY child that is
     `position: absolute` AND `contenteditable="false"` should also
     contain at least one in-flow text container (<p>/<br>) by
     construction. If you can't guarantee that from the markup,
     inject a placeholder on the JS side before any user interaction
     can land in the editable region.

### R-bookwriter-chapter-numbering-gap-after-delete. Delete left a gap; new chapter took max+1 from the gap
Date: 2026-04-24
Was: Deleting a middle chapter (e.g. chapter 6 from 1..8) left the
     remaining numbers as 1,2,3,4,5,7,8 — a visible gap. The next
     "+ new chapter" then computed max(chapter_number)+1 = 9 from
     the gappy state, producing a chapter number that "continued
     from past" instead of slotting into the empty position.
Root cause: api_bookwriter_chapter_delete in views_api_chapter.py
     only set is_active=False on the row. No shift was applied to
     remaining chapters' chapter_number / sort_order. The
     api_bookwriter_book_chapter_create endpoint correctly excluded
     deleted rows from its Max(chapter_number) aggregate, but the
     remaining active rows themselves still carried their original
     numbers, so Max returned the highest pre-deletion number.
Fix (going forward):
     `Chapter.objects.filter(book=…, is_active=True,
        chapter_number__gt=deleted_chapter_number).update(
            chapter_number=F('chapter_number') - 1,
            sort_order=F('sort_order') - 1,
            updated_at=saved_at)`
     applied immediately after the soft-delete in
     views_api_chapter.py:api_bookwriter_chapter_delete.
Fix (one-time for pre-existing gaps):
     ```sql
     WITH renumbered AS (
       SELECT bookwriter_chapter_id,
              ROW_NUMBER() OVER (
                  PARTITION BY link_bookwriter_coll_book_id
                  ORDER BY chapter_number, bookwriter_chapter_id
              ) AS new_number
         FROM [bookwriter].[chapter] WHERE is_active = 1)
     UPDATE c SET c.chapter_number = r.new_number,
                  c.sort_order     = r.new_number,
                  c.updated_at     = SYSDATETIME()
       FROM [bookwriter].[chapter] c
      INNER JOIN renumbered r ON c.bookwriter_chapter_id =
                                 r.bookwriter_chapter_id
      WHERE c.chapter_number != r.new_number
         OR c.sort_order     != r.new_number
     ```
     Verified end-to-end on real data with savepoint + rollback,
     then applied. Book #4 went {1,2,3,4,5,7} -> {1,2,3,4,5,6}.

### R-bookwriter-page-break-never-shrinks. Empty chapter showed N empty pages after switching from a long chapter
Date: 2026-04-24
Was: Switching from a long chapter (3+ pages of content) to a
     short or empty chapter showed the empty chapter with N empty
     A4 page outlines instead of 1.
Root cause: recomputePageBreakMarkers in
     modules/bookwriter-page-breaks.js measured content height with
     proseElement.scrollHeight, then set proseElement.style.minHeight
     to round up to the next full A4 page. scrollHeight returns
     max(actualContentHeight, minHeight) — so once minHeight grew,
     scrollHeight was forever inflated by the overlay's own previous
     output. The contenteditable prose element survives chapter
     switches (only innerHTML is replaced), so leftover style.minHeight
     persisted into the new empty chapter and the page count locked
     in at the previous high value.
Fix: switched the measurement to
     findLastContentBottomPxRelativeToProse(proseElement) — a helper
     that already existed for the end-of-chapter ornament. It walks
     proseElement.children backwards, skips the overlay, skips the
     empty <p><br></p> placeholder (height < 12), and returns the
     bottom edge of the last real-content child. Bypasses the
     min-height padding entirely.
     The existing `if currentMinHeightPx !== fullPageHeightPx`
     write-guard prevents a ResizeObserver feedback loop when
     min-height shrinks: the second recompute pass sees the same
     content bottom and exits without writing.

### R-bookwriter-rail-flex-overflow-not-scrolling. Book-info pushed off-screen / squeezed chapter list
Date: 2026-04-24
Was: Two related symptoms across attempts:
     (1) Long chapter list (8+ chapters) pushed the book-info block
         OUT of the rail's visible area — book-info stays anchored
         in the rail per CSS but its bottom got clipped by the
         rail's overflow:hidden + max-height.
     (2) Adding flex-shrink: 0 to .bookwriter-book-info to fix (1)
         caused the OPPOSITE bug: book-info's natural height (~700px
         for stats + 4 metadata fields + synopsis textarea) ate the
         rail and squeezed the chapter list to nothing.
Root cause:
     For (1): .bookwriter-chapters had `flex: 1; overflow-y: auto`
     but NO `min-height: 0`. Default flex-item min-height is `auto`,
     which forbids shrinking below intrinsic content size. So the
     chapter list refused to shrink, pushing book-info past the
     rail's max-height.
     For (2): flex-shrink: 0 alone made book-info incompressible at
     its natural ~700px. With no upper bound, chapters got squeezed.
Final fix (architectural — see Book Settings drawer change-log):
     Moved the 4 metadata fields out of inline book-info into a
     slide-from-right drawer (.bookwriter-book-settings-panel),
     leaving book-info compact (~120px: title + author + stats +
     gear button). With book-info naturally short:
       - .bookwriter-chapters keeps `flex: 1; min-height: 0;
         overflow-y: auto` for proper internal scroll
       - .bookwriter-book-info keeps `flex-shrink: 0` (safety, no
         longer needed in practice but cheap)
       - The intermediate `max-height: 50%; overflow-y: auto` hacks
         on book-info were REMOVED — the drawer made them obsolete
Lesson: any flex-column item with `overflow: auto/scroll` MUST also
     have `min-height: 0` (column) or `min-width: 0` (row) for the
     scrollbar to actually engage. Default `min-height: auto` is the
     trap. This is a well-documented CSS gotcha.

### R-bookwriter-pdf-export-weasyprint-tesseract-dll. WeasyPrint loaded wrong libgobject DLL from Tesseract
Date: 2026-04-23
Was: Bookwriter book-as-PDF export shipped using the Edge-headless
     renderer (tools/pdf_export_utils.render_html_string_to_pdf_bytes).
     Output had two problems even after the renderer ran cleanly:
       1. Embedded /media/upload/bookwriter/chapter/*.png images were
          missing from every chapter page (Edge running in headless
          file:// mode refuses cross-origin loads of /media URLs even
          with --allow-file-access-from-files; the only reliable fix
          is rewriting /media + /static to absolute file:// URIs in
          the HTML before printing, which adds fragile path-mapping
          code per app).
       2. No PDF outline / bookmarks for chapters. Edge headless does
          not emit an outline from HTML at all. Fallback was a
          pypdf-based text-search post-pass (already coded as
          add_outline_bookmarks_to_pdf_bytes) that scans each page's
          text for the chapter title — works but fragile against text
          extraction quirks.
     Decision: switch bookwriter to WeasyPrint, which reads CSS
     `bookmark-level` + `bookmark-label: content()` natively (PDF
     outline emitted directly into the catalog), AND resolves /media
     and /static URLs against a `base_url` argument with no manual
     rewriting needed, AND uses Pango for real Bengali Unicode
     shaping. Wrote render_html_to_pdf_bytes_via_weasyprint() in
     tools/pdf_export_utils.py as a sibling renderer to the
     Edge path. Bookwriter view tries WeasyPrint first, falls back
     to Edge-headless on ImportError so export always succeeds.
     User installed WeasyPrint 68.1 + GTK3 Runtime. First
     `import weasyprint` raised:
       OSError: cannot load library
       'C:\Program Files\Tesseract-OCR\libgobject-2.0-0.dll':
       error 0x7e
     Root cause: WeasyPrint loads its native deps via CFFI
     (Pango / GLib / GDK-PixBuf / Cairo), which uses Windows'
     standard DLL-search order — and Windows' standard order
     walks every directory listed in `PATH` from left to right
     and returns the FIRST hit. Tesseract-OCR ships its own
     stale `libgobject-2.0-0.dll` (vendored alongside its OCR
     binaries) and was earlier in PATH than the freshly-installed
     `C:\Program Files\GTK3-Runtime Win64\bin`. CFFI grabbed the
     Tesseract copy, which is built against a different GLib ABI,
     and the load failed before WeasyPrint could even initialise.
Fix: User reordered system PATH so `C:\Program Files\GTK3-Runtime
     Win64\bin` appears BEFORE `C:\Program Files\Tesseract-OCR`.
     After reorder + new shell, `python -c "import weasyprint;
     print(weasyprint.__version__)"` returned `68.1` cleanly and
     the bookwriter export round-tripped — user confirmed
     "i tested, i can see the picture" and bookmarks now appear in
     Adobe / Edge / Foxit sidebars. Cache v851 → v857.
Lesson: Any future WeasyPrint consumer on a Windows dev machine
     that ALSO has Tesseract installed will hit this exact failure
     unless GTK3-Runtime\bin is ordered before Tesseract-OCR in
     PATH. Document this as a one-line PATH check before debugging
     anything else when `import weasyprint` fails on Windows.
     Linux / Mac don't have this problem (separate package
     namespaces). The graceful Edge-headless fallback in
     bookwriter/views_api_book.py:api_bookwriter_book_export_pdf
     means a missing / broken WeasyPrint never breaks the export
     — only the bookmarks downgrade from CSS-native to
     pypdf-text-search.

### R-bookwriter-bangla-input-conjunct-cursor. Cursor stuck mid-word for conjunct Bengali
Date: 2026-04-23
Was: Wired BanglaInput into the bookwriter chapter title + prose +
     full-screen focus surfaces (all contenteditable). Simple
     Bengali words like "সমাজ" (4 codepoints, 4 visible graphemes
     == 1:1) typed and committed correctly. But conjunct words
     like "রাষ্ট্র" (7 codepoints, 2 visible graphemes — র + া
     forms the first cluster, ষ + ্ + ট + ্ + র the second) caused
     the cursor to land between the two clusters after the auto-
     pick instead of at the end. The user then had to type more
     words to "release" the cursor.
     Root cause: the contenteditable replacement code used
     manual Range API:
       range.deleteContents();
       var insertedTextNode = document.createTextNode(newText);
       range.insertNode(insertedTextNode);
       caretRange.setStart(insertedTextNode, insertedTextNode.length);
     `textNode.length` returns UTF-16 code units (= codepoints
     for BMP chars). For "রাষ্ট্র", length = 7. setStart at
     offset 7 SHOULD be the end of the text node. But Selection
     in contenteditable resolves the position relative to GRAPHEME
     boundaries when displaying the caret, and 7 codepoints inside
     a 2-grapheme cluster places the visual caret between the
     graphemes instead of after them.
Fix: Replaced manual deleteContents/insertNode/setStart with
     document.execCommand('insertText', false, text) after first
     selecting the range to replace. execCommand is deprecated but
     it's the only API that handles grapheme-cluster cursor
     placement correctly across browsers, AND fires the input
     event natively (so we no longer need to dispatch one
     manually). Single-node replacement works for all word types.
     Also flagged: the .bookwriter-prose drop cap (::first-letter
     at 4.5em float:left) was an aggravating factor — it produced
     a giant floated character that further confused cursor
     positioning during edit. Scoped the drop cap to
     :not(:focus-within) so it only renders when the prose isn't
     actively being edited. Visual flourish for non-editing
     contexts (and the user confirmed they prefer it that way:
     "it is working but not showing while typing — that is fine").
Cache v838 -> v841.

### R-bookwriter-autosave-pyodbc-ntext. Doubled chapter content lost on revisit
Date: 2026-04-22
Was: User typed a chapter, then doubled it by paste. On chapter re-open
     the doubled content was gone — only the original survived. Server
     traceback showed `Cannot convert to text/ntext or collate to
     'Latin1_General_100_CI_AS_SC_UTF8'` on
     api_bookwriter_chapter_autosave's chapter.save(update_fields=[
     'chapter_text_html', 'chapter_text_plain', ...]) call. Same
     pyodbc bug as memory/troubleshooting.md §5b — pyodbc auto-promoted
     the long string to SQL_WLONGVARCHAR (ntext on the wire), which the
     UTF-8 collation rejects at the parameter-binding layer. CAST in
     the query can't save it. JS .catch() swallowed the 500 silently
     (saveChip showed "offline · will retry" briefly), DB never
     received the new content, revisit pulled the OLD content.
Fix: Replaced the .save(update_fields=[...]) with a raw pyodbc cursor
     in views_api_chapter.py:api_bookwriter_chapter_autosave. Pattern:
       import pyodbc
       django_db_connection.ensure_connection()
       raw_cursor = django_db_connection.connection.cursor()
       raw_cursor.setinputsizes([
           (pyodbc.SQL_WVARCHAR, 0, 0),  # chapter_text_html
           (pyodbc.SQL_WVARCHAR, 0, 0),  # chapter_text_plain
           None, None, None, None,       # int + datetimes + WHERE pk
       ])
       raw_cursor.execute("UPDATE [bookwriter].[chapter] SET ... WHERE bookwriter_chapter_id = ?", [...])
       django_db_connection.connection.commit()
     Mirrored values to the in-memory chapter instance so callers below
     see consistent state. Verified with a 9107-char throwaway-chapter
     happy-path test — UPDATE returned 200 and the row carried 9107
     chars in chapter_text_html post-roundtrip.
Why .catch() didn't surface this: the autosave is fire-and-forget for
     UX reasons (don't block typing on a flaky network). Future hardening:
     escalate persistent autosave failures into a visible "save failed —
     copy your text now" inline banner instead of a transient saveChip.

### R-bookwriter-prose-image-feature. Drag-drop inline images
Date: 2026-04-22
Was: User wanted to drop an image into the manuscript prose, drag it to
     reposition, resize without distortion, text wrapping around it.
     None of these existed.
Fix: Built as a self-contained module + matching backend. Six pieces:
     1. core/utils.py:sanitize_user_html now accepts additive
        extra_allowed_tags / extra_allowed_attrs_by_tag /
        extra_allowed_void_tags / attr_value_validators_by_tag_attr.
        Default behaviour unchanged for every other caller.
     2. bookwriter/views_api_helpers.py:sanitize_chapter_prose_html
        — opts in <img> with src/alt/class/style/data-float, each
        constrained by per-attr validators (src must start with
        /media/upload/bookwriter/chapter/, style must match
        `width: Npx`, class restricted to .bookwriter-prose-image*
        family, data-float in {left,right,center,none}).
     3. bookwriter/image_uploads.py — Pillow.verify() validates real
        image bytes (rejects spoofed extensions + image bombs); SVG
        explicitly rejected (XSS via <foreignObject>); 5 MB cap;
        UUID filename to prevent collisions and path traversal.
     4. POST /bookwriter/api/chapter/<id>/image/upload/ — owner-checked,
        returns the public /media URL.
     5. modules/bookwriter-prose-image.js — drag-drop on prose,
        caret-positioning at drop point via document.caretPositionFromPoint
        (with caretRangeFromPoint fallback for Safari), click-cycle

### R-scroll. Page jumps to top on nav / form-action / reload — two mechanisms
Date: 2026-04-21 (consolidates R0 from 2026-04-16)

Two distinct ways the scroll resets to top. Diagnose first (DevTools Network):
full document reload = mechanism A; XHR + `<main>` swap = mechanism B.

Mechanism A — full reload (`window.location.search/href = ...`, plain `<a href>`)
  Browser default scroll behaviour on a fresh load is (0, 0) UNLESS the URL has
  a fragment (`#anchor`) pointing at an in-DOM id.
  Fix: add `id="..."` to the in-DOM element the user was focused on, append
  `#same-id` to every URL the JS or template emits. Build URLs explicitly as
  `pathname + '?' + params + '#anchor'` — `window.location.search = ...` DROPS
  any hash. Verified: Quiz Panel review queue (`id="quizadmin-review-card"` +
  `#quizadmin-review-card` on every approve/reject/prev/next URL).

Mechanism B — SPA navigation (sidebar clicks → spa-navigation.js)
  spa-navigation.js Step 8 used to call `window.scrollTo(0, 0)` on every
  forward nav. Wrong for any long-scroll page (Inkwell manuscript, feed,
  bookmarks). Reported 10+ times before being properly committed + documented.
  Fix: per-URL save/restore in spa-navigation.js —
    Step 0 (leaving): `sessionStorage['spa_scroll:' + pathname] = scrollY`
    Step 8 (arriving): if `sessionStorage['spa_scroll:' + destPath]` exists,
                        restore it; else scroll to top.
  Universal across every sidebar link. Sidebar's OWN scrollTop preservation
  (sidebar-navigation.js sessionStorage) was already in place; this is the
  body/window equivalent.

  Sub-bug B2: clicking the SAME active sidebar link twice still scrolled to
  top. Click handler early-returned on `url === pathname` without calling
  `event.preventDefault()`, so the browser's default same-href reload kicked
  in → fresh page load → scroll (0, 0). Fix: preventDefault on the same-
  pathname branch too. True no-op now.

Files: spa-navigation.js (Step 0 + Step 8), seo/views.py (cache bump).

Verification (browser, do every time):
  1. Scroll mid-page on home feed.
  2. Click কলম sidebar → /bookwriter/ at top (first visit).
  3. Scroll mid-Inkwell.
  4. Click any other sidebar item → leaves bookwriter.
  5. Click কলম again → MUST restore step 3 scroll position.

Why not Turbo: project already has working custom PJAX SPA. Swapping engines
would force re-validating cleanup registry / Tom Select / Quill destroy paths
across every page. ~10-line save/restore inside the existing SPA gives the

### R-bookwriter-inline-style-pattern. Inline style="background:linear-gradient(...)" violated Gate 4 #2 even for DB-driven values
Date: 2026-04-20
Was: Bible avatars had `style="background:linear-gradient(135deg, {{ hex_1 }}, {{ hex_2 }});"`.
     Both the gradient angle (135deg) and the design (linear-gradient + fallback)
     lived inline. The naive read of "dynamic values are exempt" allowed the
     whole inline block, mixing design and data.
Root cause: We treated the entire gradient as one indivisible "dynamic value".
     In reality, only the two hex values are dynamic. The angle, fallback
     colours, and gradient syntax are static design.
Fix: Adopted the CSS-custom-property injection pattern:
       CSS:    .bookwriter-bible-avatar {
                  --bookwriter-bible-avatar-hex-1: var(--ochre);    /* fallback */
                  --bookwriter-bible-avatar-hex-2: var(--accent);   /* fallback */
                  background: linear-gradient(135deg,
                    var(--bookwriter-bible-avatar-hex-1),
                    var(--bookwriter-bible-avatar-hex-2));
                }
       Template: style="--bookwriter-bible-avatar-hex-1: {{ hex_1 }};
                        --bookwriter-bible-avatar-hex-2: {{ hex_2 }};"
       JS:      el.style.setProperty('--bookwriter-bible-avatar-hex-1', hex1);
                el.style.setProperty('--bookwriter-bible-avatar-hex-2', hex2);
     Inline still exists but carries data only (no design, no syntax).
     Now codified in CLAUDE.md Gate 4 #2.
Lesson: When you find yourself emitting inline `style="..."`, ask "is the
     ENTIRE inline value dynamic, or only N specific data points?" If the
     latter, use custom properties + a CSS rule that composes them. The
     inline attribute should never carry a CSS function call (linear-gradient,
     calc, rgb, etc.) — those are design.

### R-bookwriter-classes. Bulk CSS class rename broke styling on cached browsers
Date: 2026-04-20
Was: After a bulk rename of every bookwriter CSS class to add the
     `bookwriter-` prefix, the inkwell page lost its drop-cap (first
     letter big + accent color) and other visual cues. The CSS file
     was correct, the templates were correct, but the SERVICE WORKER
     was still serving the previous CSS where the selector was `.prose`,
     not `.bookwriter-prose`.
Root cause: Two-fold. (a) The bulk-rename script only rewrote class
     names in CSS rules, in HTML `class="..."` attributes, and in JS
     `classList.*` / `.className` / dot-prefixed selector strings.
     It MISSED bare class-name strings inside JS string concatenations
     (e.g. `' is-resolved'` in a ternary), inside JS `innerHTML` template
     strings (`<div class="margin-notes-empty">`), and inside object-
     literal options whose key implied a class (e.g.
     `confirmingClass: 'confirming'`). (b) Service worker cache was
     not bumped after the CSS rename.
Fix: Wrote a context-aware pass-2 script that handles innerHTML class=
     attrs, classList.* calls including leading-whitespace forms,
     `.className =/+=` full RHS (including ternaries), and object-literal
     properties whose key ends in `Class`. Then bumped CACHE_NAME from
     v759 → v760 + ran collectstatic. User had to hard-reload once
     (Ctrl+Shift+R) for the service worker to refresh.
Lesson: When doing a bulk CSS class rename, the JS rewrite MUST cover:
     (1) classList.add/remove/contains/toggle/replace string literals,
     (2) `.className = / +=` assignments INCLUDING the entire RHS
     expression (ternaries, concats),
     (3) innerHTML / template string `class="..."` attributes,
     (4) object-literal property values whose KEY name ends in `Class`,
     (5) dot-prefixed selector strings inside querySelector etc.
     Bumping CACHE_NAME + collectstatic is non-negotiable after any
     CSS edit, even pure renames.

### R28. Hardcoded is_staff_user=False breaks staff edits on legacy data
Date: 2026-04-17

### R29. CSS custom-property theming via [data-reader-theme="..."]
Date: 2026-04-17
Was: Book reader v2 needed light / sepia / dark themes that switch
instantly without page reload AND cascade onto every element inside the
reader (background, text, borders, accent, muted text). Class swapping
(adding/removing .theme-dark) requires every styled element to know about
the theme class — fragile.
Pattern: Set CSS custom properties on the root reader element gated by
[data-reader-theme="sepia"] and [data-reader-theme="dark"] attribute
selectors. The base rule defines all variables (--reader-bg, --reader-text,

var(--reader-*), so swapping the data attribute on the root re-cascades
the entire palette.
Bonus: same pattern works for font-size — [data-reader-font-size="..."]
gates a single rule that sets font-size on .reader-pane.

### B7. Bookwriter — _strip_html_to_plain leaves quotes/semicolons untouched
General pattern: bookwriter's _strip_html_to_plain regex is "<[^>]+>" — it
only strips HTML tags. For free-form text fields (entry_role, entry_name,
margin notes, comment_text) this is fine because the resulting string is
rendered via Django's autoescape (not |safe). NEVER use _strip_html_to_plain
output inside a style="" / data-attr / inline-onclick context — those need
either escapejs / escape filters or a strict pattern validator.
