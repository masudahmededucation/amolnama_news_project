================================================================
PORTAL APP — Architecture & Features
App: site_apps/portal/
URL prefix: /portal/
================================================================

## Overview
User profile management hub. Edit public profile, personal details, contact, address, settings.
Avatar upload with Cropper.js.

## Models
None — uses UserProfile + Person from user_account app.

## URL Patterns

| URL                          | View                  | Name                   |
|------------------------------|-----------------------|------------------------|
| /portal/                     | home                  | portal:home            |
| /portal/profile/             | profile_redirect_view | portal:profile         |
| /portal/profile/public/      | profile_public_view   | portal:profile_public  |
| /portal/profile/personal/    | profile_personal_view | portal:profile_personal|
| /portal/profile/contact/     | profile_contact_view  | portal:profile_contact |
| /portal/profile/address/     | profile_address_view  | portal:profile_address |
| /portal/profile/settings/    | profile_settings_view | portal:profile_settings|
| /portal/api/avatar/upload/   | api_avatar_upload     | portal:api_avatar_upload|

## Views

### profile_public_view
Editable fields: display_name, username_handle (@username), bio (short), about (long), profile_location, profile_website_url.
- @username auto-generated from display name if not set
- Username uniqueness check via generate_username_handle() in core/utils.py
- @ prefix shown in UI via CSS wrapper

### profile_personal_view
Editable fields: first/last name (en/bn), alias (en/bn), father/mother names, DOB, gender, religion, NID.

### profile_contact_view
Editable fields: phone (country code + number), email.

### profile_address_view
Editable fields: home address fields.

### api_avatar_upload
POST with multipart file. Validates image type/size. Stores in media/avatars/. Updates UserProfile.avatar_file_path.

## Key Functions

### Views (views.py)
- home() — landing page, redirects authenticated users to profile
- profile_redirect_view() — redirects /portal/profile/ → /portal/profile/public/
- profile_public_view() — GET: render form with current data. POST: validate + save display_name, username_handle, bio, about, location, website to UserProfile
- profile_personal_view() — GET/POST: edit Person fields (names en/bn, DOB, gender, religion, NID)
- profile_contact_view() — GET/POST: edit Phone + Email records
- profile_address_view() — GET/POST: edit PersonAddress
- profile_settings_view() — account settings management

### APIs (views_api.py)
- api_avatar_upload() — accepts multipart file, validates image type (JPEG/PNG/WebP) and size (<5MB), stores in media/avatars/<user_profile_id>/, updates UserProfile.avatar_file_path

### Utilities Used
- generate_username_handle(display_name) from core/utils.py — auto-generates @username, ensures uniqueness

## Troubleshooting
- @username auto-generated silently on first profile save — no warning needed
- Avatar upload uses Cropper.js for client-side crop before upload
- Profile sidebar highlights current section via URL matching
- display_name updates propagate everywhere via user_display_name context processor
- Person model: import as `from user_account.models import Person` (not person.models)

## Admin Dashboard Views

### content_dashboard_view
Shows a unified list of all content (blog + newshub) ordered by newest
first. Each row has: content type badge, title, slug, detail URL link,
publish status, created_at. Staff can toggle publish via
api_content_toggle_publish.

Data assembled by `_collect_content_dashboard_items()` — see Phase 3b
refactor below.

### analytics_dashboard_view
Shows aggregated platform metrics: total posts, total engagements,
active users in last 30 days, content type breakdowns.

### moderation_queue_view
Shows flagged content across all types. Staff can Approve (clear flags)
or Reject (soft-delete) via api_moderation_approve /
api_moderation_reject.

### composer_placeholders_view
Admin panel for the composer placeholder system (used by the post wall
"What's on your mind?" prompts). Staff can Add, Toggle active, Feature
(for N minutes), Delete placeholders.

## Phase 3b Refactor: _collect_content_dashboard_items

Commit 569110d (2026-04-11) replaced an 80-line hand-maintained
`content_sources` list with a 15-line loop over the unified content
type registry.

### Old shape (pre-2026-04-11)
```python
content_sources = [
    {'content_type': 'poem', 'id_attribute': '...', 'title_attribute': 'poem_title_bn',
     'slug_attribute': 'poem_slug', 'url_prefix': '/bangla-kobita-gaan/',
     'status_code_attribute': 'poem_status_code', ...},
    {'content_type': 'stories', 'id_attribute': '...', 'title_attribute': 'story_title_bn', ...},
    # 5 more entries with the same manual per-type knowledge
]
```
Each entry duplicated per-type field names that were already known to
content_type_registry.

### New shape (post-Phase 3b)
```python
from amolnama_news.site_apps.content.content_type_registry import (
    iter_all_specs, get_model_class,
    get_title, get_slug, get_detail_url, get_is_published, get_pk,
)

for code, spec in iter_all_specs():
    model_class = get_model_class(code)
    if model_class is None:
        continue
    for instance in model_class.objects.all().order_by('-created_at')[:50]:
        content_items.append({
            'content_type':         code,
            'content_type_label':   spec['badge'],
            'content_type_color':   spec['color'],
            'content_id':           get_pk(instance),
            'content_title':        get_title(instance) or '(no title)',
            'content_slug':         get_slug(instance),
            'content_url':          get_detail_url(instance),
            'is_published':         get_is_published(instance),
            'created_at':           instance.created_at,
            'created_at_formatted': _format_created_at(instance.created_at),
        })
```

**Newshub handled separately** in a second loop because PubArticle has a
split schema (pub_article_headline_bn, pub_article_slug,
pub_article_id) that doesn't fit the registry. Until PubArticle is
unified, the special case stays.

**Output shape unchanged** — the template still receives the same dict
keys, so content-dashboard.html was untouched. Adding a new blog
content type to the dashboard now requires ZERO changes here — just
add a registry entry in content/content_type_registry.py.

## Helper Functions (views.py / views_api.py)

- `_format_created_at(created_at_value)` — format timestamp for display,
  returns empty string if None
- `_collect_content_dashboard_items()` — main collector described above
- `_parse_json_body(request)` (views_api.py) — parses JSON POST body,
  returns (data, error_response_or_None). Used by all admin POST APIs.
