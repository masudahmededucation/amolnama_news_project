What auto-scales on new hardware (zero code change)
Ollama: detects GPU (CUDA/ROCm) automatically. Plug in an RTX 4090 → llama goes GPU → ~10x faster generation. No config.
mDeBERTa NLI + sentence-transformers: use torch. If CUDA visible, they auto-use GPU. Otherwise CPU.
Ollama CPU threads: uses all cores by default.
SQL Server connection pool: scales with DB server config, independent of our Python.
What does NOT auto-scale
Chunk processing loop (ai_generator.py:823) is strictly sequential — one chunk at a time, regardless of core count. On a 32-core server we'd still use ~1 core of wall-clock for generation.
Tesseract OCR — one page at a time, CPU-bound.
Embedding saves — one DB round-trip per question.
Can we make it automatic?
Yes — a single mastermind/runtime_profile.py module introspects hardware at startup and picks defaults:


import os, psutil, torch

WORKERS = max(1, os.cpu_count() - 1)          # 3 on laptop, 31 on server
USE_GPU = torch.cuda.is_available()            # False on laptop, True on server
OCR_PARALLELISM = max(1, os.cpu_count() // 2)  # I/O + CPU mix
EMBEDDING_BATCH_SIZE = 16 if not USE_GPU else 128
OLLAMA_CONCURRENCY = 1 if not USE_GPU else 4   # GPU handles concurrent requests cleanly
Engine imports constants from there. Laptop: modest. Server: aggressive. Zero manual config.

Recommendation
Not now. Premature. Current bottleneck is llama3.2:3b on CPU (~60-180s per call) + Bengali JSON parse failures. Even with parallelism we'd still be bottlenecked on the LLM itself.

When to build the runtime_profile module: after we deploy to actual production hardware AND the bottleneck analysis shows parallelism matters. Estimated ~2 hours of work when it's needed. I'd rather leave the code simple + sequential until there's a real server to test against.
